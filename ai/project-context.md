# Project Context

Проект: внутренний macOS-инструмент для производства медицинских SEO-статей клиники.

## Что это за проект

Персональная (single-user) писательская студия для одного редактора. Ведёт каждую тему статьи по пайплайну от брифа до публикации: автоматический сбор семантики (Wordstat + ИИ-воронка), задача читателя (поисковый интент), генерация и доработка текста несколькими ИИ-агентами, проверки (SEO, фактчекинг, вычитка), генерация иллюстраций, финальная публикация в Google Docs. Цель — производить медицинские SEO-статьи быстро и с контролем качества и фактов.

Статус: базовый пайплайн реализован целиком (от брифа до публикации), идёт добавление и доработка отдельных фич. Реализованы все под-проекты исходного плана 1–9 (Фундамент, База знаний, Ядро генерации, Проверяющие этапы, Редактор промтов этапов, ИИ-роли и блоки контекста, Структура, Генерация изображений, Публикация в Google Docs — очередь из плана убрана насовсем, см. `ai/decisions.md`). Поверх плана добавлены: продуктовые блоки и скиллы, словарь мягких подсказок, семантический AI-агент и автосбор семантики через Wordstat с журналом-воронкой, этап «Анализ и обучение» (рекомендации по промтам), персональные защищённые дефолты промтов, карта задачи читателя, загрузка иллюстраций на Google Drive + раздел «Техническая информация».

Реализованные @Model-сущности (SwiftData, полный список — в `SEOContentCreatorApp.swift`): Topic, ReaderIntent, KnowledgeNode, ArticleVersion, GenerationJob, StageTemplate, ContextBlock, AIRole, GeneratedImage, ImageStylePreset, ImagePromptTemplate, ExternalDocument, EditorDictionary, SkillPreset, SemanticKeyword, PublishedSitePage, ProductBlock, ForbiddenPhrase, PersistedRemark, PromptRecommendation, SemanticStopWord, SemanticQueryMask, SemanticFunnelEntry (ArticleType/NodeType/PipelineStage и т.п. — это enum'ы, не @Model). Любая новая @Model-сущность должна быть зарегистрирована и в `SEOContentCreatorApp.swift`, и во всех тестовых `ModelContainer`, использующих ту же схему. Полная спека — `docs/superpowers/specs/2026-05-19-content-system-redesign-design.md`.

## Стек

Нативное macOS-приложение на **SwiftUI** (решение от 2026-05-21, см. `ai/decisions.md`). Веб-обёртка отклонена (нужен только macOS). Потребуется Xcode. Skill `frontend-design` (веб) напрямую не применяется — UI проектируется макетами и реализуется в SwiftUI.

## Как запустить локально

Открыть `SEOContentCreator/SEOContentCreator.xcodeproj` в Xcode → Cmd+R.

## Как собрать

`cd SEOContentCreator && xcodebuild -scheme SEOContentCreator -destination 'platform=macOS' build`

## Как запустить тесты

В Xcode — Cmd+U (надёжный путь).

Из CLI команда `xcodebuild test` зависает при старте раннера (известная проблема окружения, см. память `xcodebuild-test-runner-hang`). Рабочий обход из CLI — сборка и прогон раздельно, с ограничением до unit-таргета:

`cd SEOContentCreator && xcodebuild build-for-testing -scheme SEOContentCreator -destination 'platform=macOS' && xcodebuild test-without-building -scheme SEOContentCreator -destination 'platform=macOS' -only-testing:SEOContentCreatorTests`

Только компиляция (быстрая проверка, ловит и ошибки в тестах): `xcodebuild build-for-testing -scheme SEOContentCreator -destination 'platform=macOS'`.

## Инварианты проекта

Правила, которые нельзя ломать.

- Single-user (один редактор, без ролей и блокировок).
- Google Docs — только финальная публикация, без двусторонней синхронизации.
- Версии текста не удаляются, только архивируются.
- Публикация — только по явному действию (нет авто-публикации).
- ИИ не переписывает текст молча: проверки показывают предложения, человек принимает.
- Проверяющие ИИ-роли отделены от автора (разные мандаты, вход, знания).
- Пайплайн нежёсткий: любой этап можно запустить в любой момент.
- Ручная правка — сквозная, доступна на любом этапе.
- Черновик нельзя запустить без названия, типа статьи и направления.

## Хрупкие зоны

- На этапе реализации обратить внимание на: согласованность единой ленты версий, фактчек по Базе знаний, отсутствие обратной синхронизации с Docs.
