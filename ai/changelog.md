### 2026-07-03 — Массовая вставка списка запросов в окно "Семантика"

- Change: [SemanticsEditorSheet.swift](SEOContentCreator/SEOContentCreator/Views/SemanticsEditorSheet.swift)
  получил кнопку "Вставить список..." рядом с однострочным полем добавления. Открывает sheet с
  многострочным полем — можно вставить список, скопированный из Yandex Wordstat. Каждая строка
  парсится: если в конце строки через таб идёт число — это частотность, остальное — текст
  запроса; иначе вся строка — запрос без частотности. Дубликаты (с существующими и внутри самого
  вставляемого списка, без учёта регистра) пропускаются, показывается сообщение "Добавлено N,
  пропущено M".
- Impact: пользователь может разом добавить весь список ключевых запросов из Wordstat вместо
  добавления по одному.
- Manual checks: `xcodebuild build-for-testing` прошёл успешно; ручная проверка в живом UI не
  делалась отдельно от Release-сборки ниже.

### 2026-07-03 — Фикс тормозов интерфейса при длинной генерации + Release-переустановка

- Change: [MarkdownBlocksView.swift](SEOContentCreator/SEOContentCreator/Views/MarkdownBlocksView.swift)
  разбирал Markdown всей текущей статьи (`MarkdownDocParser.parse`, O(длины текста), без кэша)
  заново при каждой перерисовке экрана. Так как `StageExecutor.streamingText` (@Observable)
  обновляется на каждый токен генерации, а `SideBySideView.leftColumn` рендерит неизменную
  "текущую версию" через `MarkdownBlocksView` даже во время стриминга, вся уже готовая статья
  перепарсивалась с нуля на каждый токен. Замер (`swift bench.swift` на статье ~33 тыс. символов):
  ~6.7 мс на один разбор → до ~56 секунд лишней работы на генерации в 8000 токенов, причём на том
  же главном потоке, что читает ответ от OpenAI — это и объясняет видимое замедление генерации
  по мере роста текста. Исправлено: результат разбора теперь кэшируется в `@State`, пересчёт —
  только когда сам текст меняется (`.task(id: text)`), а не при каждой перерисовке.
- Impact: Скорость обновления UI и восприятие скорости генерации больше не деградирует
  пропорционально объёму уже накопленного текста статьи. Логика самого парсера и модель данных
  не менялись.
- Manual checks: `xcodebuild build-for-testing -scheme SEOContentCreator -destination 'platform=macOS'`
  — TEST BUILD SUCCEEDED. Release-сборка (`xcodebuild -scheme SEOContentCreator -configuration
  Release build`, BUILD SUCCEEDED) установлена в `/Applications/SEOContentCreator.app` (старая
  версия удалена, новая запущена — процесс стартовал без падения). Реальный сквозной прогон
  длинной генерации на статье с большим объёмом уже готового текста, с замером на глаз, что
  скорость появления токенов не деградирует — не выполнен в этой сессии; корневая причина
  подтверждена измерением стоимости парсинга, а не профилировщиком в живом приложении.

### 2026-07-03 — Исправлен краш при удалении узла Базы знаний, использованного темой

- Change: Найден и исправлен баг: `Topic.direction`/`Topic.doctor` (`Models/Topic.swift`) не имели `deleteRule` и обратной связи (`inverse`), поэтому при удалении `KnowledgeNode`, на который ссылалась тема, SwiftData не обнуляла ссылку — тема оставалась с «битой» ссылкой на несуществующий узел. При следующем запуске `ContentPlanView` падала с крашем при попытке прочитать `direction?.title` у такой темы (SwiftData не может разрешить fault на удалённый объект). Добавлены обратные relationship-свойства `topicsUsingAsDirection`/`topicsUsingAsDoctor` в `Models/KnowledgeNode.swift` и `inverse:`/`deleteRule: .nullify` в `Topic.direction`/`Topic.doctor` — теперь SwiftData сама обнуляет ссылку у темы при удалении связанного узла. Добавлен регрессионный тест `TopicKnowledgeNodeDeletionTests` (2 теста): проверяет, что удаление направления/врача, использованных темой, обнуляет ссылку, а не оставляет её висящей. Полный набор тестов зелёный (309 → 311 passed).
- Impact: Пользователь может безопасно удалять узлы Базы знаний, даже если они использованы темой — приложение больше не падает при следующем запуске. **Обнаружено и исправлено на живых данных пользователя**: при удалении направления «Урология» тема «Тестовая новая тема» осталась с битой ссылкой, приложение перестало запускаться. Ссылка была вручную обнулена напрямую в SQLite-файле store (`UPDATE ZTOPIC SET ZDIRECTION = NULL ...`) с предварительным бэкапом и подтверждением пользователя — только у одной этой темы, остальные данные не тронуты.
- Manual checks: полный прогон тестов — 0 failures; Release-сборка установлена в `/Applications`, приложение стартует и остаётся живым несколько запусков подряд; узлы направления «Детская лор-хирургия» (включая нового врача, добавленного пользователем) и тема «Тестовая новая тема» после исправления читаются корректно.
