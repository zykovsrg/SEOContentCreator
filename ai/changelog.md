### 2026-07-23 — Автоматический сбор семантики через Wordstat + ИИ-воронку

- Change: Этап «Семантика» перестал быть ручным. Новый пайплайн: ИИ-агент (`SemanticSeedPlanner`) планирует синонимы/маски/хвосты → реальная выгрузка из Wordstat (`WordstatCloudClient` для нового API Yandex Cloud, `WordstatLegacyClient` для старого OAuth-API — переключаются в настройках через `WordstatProviderKind`) → трёхслойная очистка: правила (`SemanticRuleFilter`: минус-слова, дедуп, порог частотности, топ-100 после правил), релевантность (сужен `SemanticAgentAnalyzer`, добавлена генерация 10 длинных запросов), каннибализация (выделена в отдельный `SemanticCannibalizationChecker`). Оркестратор `SemanticCollectionRunner` ведёт журнал воронки (`SemanticFunnelEntry`/`SemanticFunnelLayer`) — по каждому запросу видно, на каком слое и почему он отсеялся; выжившие сохраняются в семантику темы сразу как «принято» (`SemanticKeywordMerger` научился не затирать решение, которое пользователь уже принял вручную). `SemanticAgentSheet` и `SemanticMockKeywordCollector` удалены, их место занял экран `SemanticFunnelView`. Справочники минус-слов и вопросительных масок редактируются во вкладке «Семантика» в «Шаблонах»; учётные данные обоих Wordstat-провайдеров — в Настройках. После того как все 14 задач плана прошли реализацию и точечное ревью по отдельности, финальный сквозной ревью всей ветки нашёл и закрыл 3 проблемы интеграции, невидимые ни одной точечной проверке: промт агента релевантности буквально просил модель всегда возвращать `frequency: null` (реальная частотность из Wordstat тихо терялась для каждого запроса — исправлено на уровне промта и подстраховано подстановкой настоящих чисел в раннере, не завися от модели); при полном провале сбора терялся `runID`, и экран воронки не мог показать уже записанные причины провала; выжившие записи в журнале всегда сохранялись с пустой причиной, теряя пояснение про длинные запросы и риск каннибализации.
- Impact: Раньше семантика собиралась вручную (вставка списка) или через агента на выдуманных запросах без частотности. Теперь редактор жмёт одну кнопку — приложение само собирает, чистит и объясняет каждое решение. Найдено и задокументировано: старый Wordstat OAuth-API отвечает ошибкой TLS-сертификата (сертификат выписан на другой хост) — это похоже на признак вывода API из эксплуатации, поэтому реализован новый API через Yandex Cloud как основной путь, а старый оставлен «на всякий случай» с понятным сообщением об ошибке.
- Manual checks: Компиляция (`xcodebuild build-for-testing`, местами с `clean`) проходила на каждом шаге у каждого субагента и была независимо перепроверена контроллером на ключевых задачах (4, 5, 10, 11 — самые рискованные по миграции данных и интеграции). Пользователь прогнал полный набор юнит- и UI-тестов через Cmd+U в Xcode: **492 теста, 1 провал** — сломался старый тест (`TemplateChipTextTests`), жёстко ожидавший ровно 4 категории в «Шаблонах», хотя эта же работа честно добавила 5-ю («Семантика»); поправлен и перепроверен компиляцией. Все тесты по семантике и Wordstat (более 60 штук, включая тесты из финального ревью) прошли реальный запуск. Ручной запуск приложения (Cmd+R) с проверкой миграции SwiftData-базы и работы экрана воронки на живых данных — единственная оставшаяся открытая проверка для пользователя; в headless-окружении сессии подтвердить это не удалось (SwiftUI `.modelContainer`/`.task` не инициализируются полностью без реальной оконной сессии). Живой запрос к Wordstat не проверялся (кроме одного разового `curl` к старому API с реальным токеном пользователя на этапе исследования — токен нигде не сохранён).
