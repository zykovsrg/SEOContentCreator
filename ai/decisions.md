### 2026-07-23 — Сбор семантики: два провайдера Wordstat за одним интерфейсом, выжившие сохраняются как «принято»

Status: active

Decision:

Реальный сбор семантики подключён через `WordstatProvider` — единый интерфейс-функцию, за которым стоят два взаимозаменяемых клиента: `WordstatCloudClient` (текущий self-service API Yandex Cloud, API-ключ + folderId) и `WordstatLegacyClient` (старый OAuth-API `api.wordstat.yandex.net`). Провайдер выбирается пользователем в Настройках (`WordstatProviderKind`, `@AppStorage`). Пайплайн: ИИ-планировщик (`SemanticSeedPlanner`) → выгрузка через выбранный провайдер → правила (`SemanticRuleFilter`, топ-100 применяется ПОСЛЕ минус-слов и порога частотности) → релевантность (`SemanticAgentAnalyzer`, теперь без cannibalization-ответственности, плюс 10 длинных запросов) → каннибализация (отдельный `SemanticCannibalizationChecker`). Оркестратор `SemanticCollectionRunner` пишет журнал воронки (`SemanticFunnelEntry`/`SemanticFunnelLayer`) — каждый запрос получает причину, почему остался или отсеялся. Настоящая частотность из Wordstat подставляется поверх ответа ИИ-агента в раннере, а не берётся из промта, — корректность не зависит от того, повторит ли модель число, которое ей уже дали.

Ключевое изменение поведения решения от 2026-07-02 («Семантика темы: решения по ключевым словам вместо списка строк»): результаты автоматического сбора (`SemanticCollectionRunner`) сохраняются сразу с `userDecision: .accepted`, а не `.pending`. Раньше агент только предлагал, пользователь решал по каждому запросу. Теперь автосбор полностью автоматический, а журнал воронки — единственная точка контроля постфактум. `SemanticKeywordMerger.merge` при этом никогда не перезаписывает решение, которое пользователь уже принял вручную (`.accepted`/`.rejected`/`.required`) — перезаписывается только `.pending`. Ручное добавление семантики (вставка списка, старый путь через `SemanticAgentSheet`, ныне удалённый) по-прежнему сохраняло `.pending` — этот путь не менялся.

Why:

Пользователь явно попросил полный автомат вместо построчного review, приняв компромисс: контроль переезжает «после факта» — в прозрачность воронки, а не в подтверждение каждого запроса. Исследование показало, что старый Wordstat OAuth-API отвечает ошибкой TLS-сертификата (сертификат выписан на `wordstat.yandex.ru`, а не на вызываемый хост) — похоже на признак вывода API из эксплуатации, официальная документация подтверждает переход на Yandex Cloud. Пользователь решил реализовать оба клиента, а не ставить всё на неподтверждённый новый API. Реальная частотность подставляется в раннере (не в промте), потому что финальный сквозной ревью всей ветки нашёл: промт буквально просил модель возвращать `frequency: null`, и без этой подстраховки частотность тихо терялась бы на каждом запуске независимо от того, что говорит модель.

Impact:

Форма ответа `WordstatLegacyClient` — неподтверждённая догадка (реальный запрос не прошёл дальше TLS-рукопожатия): использует парсер нового API, требует пересмотра если/когда старый API когда-либо реально ответит. Минус-слова и вопросительные маски (`SemanticStopWord`/`SemanticQueryMask`, редактируются в «Шаблонах») — глобальные для всех тем: слово, полезное для одной темы, может выкинуть нужный запрос у другой. Новый код, читающий/пишущий семантику через агента, должен звать `SemanticKeywordMerger.merge(..., decision:)` с явным decision, а не полагаться на дефолт `.pending`, если цель — авто-принятие.

### 2026-07-22 — Личные дефолты промтов защищены от миграций; каскадный overwrite упразднён

Status: active

Decision:

`StageTemplate`, `AIRole`, `ContextBlock` получили опциональные поля-снимки (`hasPersonalDefault` + `personalDefault*`), заполняемые через `PromptPersonalDefaultsService`. Каждое явное «Сохранить» в `StagePromptEditorView` (`saveAsPersonalDefault`) применяет состояние редактора к живому объекту И одновременно обновляет снимок — личный дефолт пользователя. «Сбросить к моему дефолту» и «Вернуть стандарт приложения» (`personalDefaultState`/`factoryState`) только загружают значения в редактор, ничего не пишут в SwiftData до следующего явного «Сохранить»; факторный сброс общей роли/блока показывает предупреждение о влиянии на другие этапы. Миграция `templatesDefaultsVersion` 8→9 удалила безусловный каскадный overwrite (описанный в решении 2026-07-02 ниже) и заменила его на `StagePromptIntentMigration.upgrade` — аддитивную, идемпотентную по маркеру `<!-- reader-intent-v1:<stage> -->` вставку по якорю (с fallback-append, если якорь не найден), которая никогда не заменяет сохранённый текст целиком, — и на однократный `captureIfMissing`, который захватывает текущее (уже апгрейженное) состояние каждого `StageTemplate`/`AIRole`/`ContextBlock` без снимка как его первый личный дефолт.

Why:

До этого решения «Сбросить к стандартному» было немедленно деструктивным (грузило заводской текст и тут же сохраняло), а миграция дефолтов безусловно перезаписывала текст каскадных этапов, контекстных блоков и мандатов ролей при каждом version bump — то есть любые правки пользователя терялись при обновлении приложения. Пользователь явно попросил защитить свои сохранённые версии промтов от будущих обновлений и одновременно сохранить возможность вернуться к заводскому варианту как к отдельному, не разрушающему действию.

Impact:

Любое будущее изменение заводского текста в `StageTemplateDefaults`/`RoleDefaults`/`ContextBlockDefaults` НЕ долетит автоматически до уже установленных инсталляций (в отличие от старого каскада) — оно становится доступно только через явное «Вернуть стандарт приложения» → «Сохранить». Если нужно доставить критичное обновление промта существующим пользователям, требуется отдельная, специально спроектированная миграция (как явно указано в дизайн-документе `2026-07-22-reader-intent-personal-prompt-defaults-design.md`), а не повторное включение старого каскада. Единственное сохранённое исключение — температура проверочных этапов (`stage.kind == .checking`) по-прежнему принудительно сбрасывается к заводскому значению при миграции (унаследовано из старого поведения, это не текст промта, а числовой параметр). Новый `@Model`-объект в `StageTemplate`/`AIRole`/`ContextBlock` с этими опциональными полями — лёгкая миграция SwiftData, существующие записи получают `hasPersonalDefault = false` и заполняются при следующем запуске `migrateTemplatesIfNeeded`.

### 2026-07-21 — Опубликованные документ и папка иллюстраций открыты «по ссылке» с правом редактирования

Status: active

Decision:

После публикации `ArticlePublisher` выдаёт документу Google Docs и папке иллюстраций на Google Диске (`<root>/Иллюстрации/№[ID] [Тема]/`) общий доступ через `GoogleDocsClient.shareWithAnyoneWithLink(fileID:role:)` — `type=anyone`, роль `writer`. Роль вынесена в константу `ArticlePublisher.linkShareRole`. Доступ папке распространяется на все файлы внутри неё автоматически (одного вызова на папку достаточно). При повторной публикации темы, у которой папка иллюстраций уже существовала до этой доработки, доступ к ней тоже открывается — доступ не остаётся привязан только к темам, опубликованным после появления функции. Ошибка выдачи доступа — предупреждение, не блокирует публикацию документа.

Why:

Пользователь явно попросил открытый доступ, чтобы контент-менеджер могла забрать картинки и открыть документ без отдельного приглашения в Google и без входа под своим аккаунтом. Роль `writer`, а не `reader`, выбрана пользователем осознанно — контент-менеджер должна иметь возможность работать с папкой, а не только смотреть.

Impact:

Любой, у кого окажется ссылка на документ или папку, может редактировать документ и удалять/добавлять файлы в папке — это принятый риск, не защита по умолчанию. Достаточно уже запрошенного OAuth scope `drive.file` (доступ только к файлам, созданным самим приложением) — расширять scope не потребовалось. Если понадобится более строгий режим («только просмотр»), это меняется заменой `linkShareRole` с `"writer"` на `"reader"`, без структурных изменений кода.

### 2026-07-09 — Модель и reasoning задаются на уровне StageTemplate

Status: active

Decision:

Модель текстовой генерации для пайплайновых этапов снова является настройкой конкретного `StageTemplate`: пользователь редактирует её в `TemplateEditorView`, значение сохраняется в `StageTemplate.modelName`, а обычный запуск этапа (`TopicWorkspaceView`) и песочница шаблона (`TemplateSandboxSheet`) используют модель из шаблона, не подменяя её глобальным `@AppStorage("openAIModel")`. Глобальная модель остаётся fallback для сценариев без собственного шаблона или для пустого значения модели.

Для GPT-5.x/o-series моделей клиент использует `max_completion_tokens` и optional `reasoning_effort`, а кастомная температура не показывается как активная настройка и не отправляется в запрос. Для legacy-моделей температура остаётся доступной и передаётся как раньше. В будущем при миграции с Chat Completions на Responses API поле `reasoningEffort` нужно маппить на `reasoning.effort`.

Why:

Пользователю нужно управлять моделью и интенсивностью мышления отдельно для каждого этапа каскада: например, держать черновик дешевле, а финальную вычитку запускать на более сильной модели. GPT-5.5 в текущей документации OpenAI управляется reasoning effort; температура для этого семейства не должна выглядеть как полезная ручка в UI.

Impact:

Новые UI-пути запуска пайплайнового этапа или песочницы шаблона не должны автоматически перетирать `StageTemplate.modelName` глобальной моделью из настроек. Если добавляется новый текстовый поток без шаблона, он может продолжать использовать `openAIModel` как fallback.

### 2026-07-05 — Дефолт стилевого пресета изображений заменён на два «стеклянных» пресета (обложка/иллюстрация)

Status: active

Decision:

`ImageStylePresetDefaults` больше не хранит один общий дефолт с брендовой палитрой клиники (`#F4F9FF`/`#007AC0`). Вместо этого — два именованных дефолта с общим "стеклянным" визуальным стилем (объект из прозрачного стекла на матовом сине-сером фоне), различающихся только композицией: «Стеклянный стиль — обложка» (объект смещён в правую треть кадра, слева место под текст) и «Стеклянный стиль — иллюстрация» (объект по центру). Оба — размер `1536x1024`, качество `high`. `ImagePromptDefaults.content(for:)` для `.cover`/`.illustration` укорочен и больше не описывает стиль/фон/свет/композицию — эти шаблоны отвечают только на вопрос «что изобразить», т.к. кнопка «Предложить через ИИ» (`ImageSubjectSuggester`) сжимает шаблон до 1–2 предложений сюжета и отбрасывает всё остальное; стиль и композиция должны жить только в пресете.

Как и роли (`AIRole`)/блоки контекста/skill-пресеты, `ImageStylePreset` сидится по принципу «таблица пуста» (`seedImageStylePresetIfNeeded`), поэтому для уже существующих инсталляций (где пресет уже есть) новые именованные дефолты не появились бы сами по себе. Добавлен явный шаг в `migrateTemplatesIfNeeded`: вставка любого дефолта из `ImageStylePresetDefaults.all`, чьё имя ещё не встречается среди существующих пресетов, без изменения/удаления уже существующих пользовательских пресетов. `StageTemplateSeeder.currentTemplatesDefaultsVersion` поднят с 6 до 7.

Why:

Пользователь попросил внедрить согласованные ранее в чате "стеклянные" промты как реальные дефолты приложения (не просто текст для копипасты). Прямая проверка в работающей debug-сборке показала, что и debug-сборка, и уже установленное в `/Applications` приложение используют один и тот же sandbox-контейнер `~/Library/Containers/com.zykovsrg.SEOContentCreator/Data/Library/Application Support/default.store` — то есть у пользователя уже есть непустая таблица `ImageStylePreset`, и без миграции новые пресеты никогда бы не появились в реальной базе (по аналогии с решением 2026-07-03 про роль `analyst`).

Impact:

Любой будущий новый дефолт в `ImageStylePresetDefaults.all` должен сопровождаться и bump'ом `currentTemplatesDefaultsVersion`, и точечной вставкой по имени в `migrateTemplatesIfNeeded` — иначе он не дойдёт до пользователей с уже существующей базой. Тесты `ImageDefaultsTests`, `ImageSeedingTests`, `StageTemplateSeederTests` обновлены под новую структуру (два дефолта вместо одного, точечная миграция).

### 2026-07-04 — StageTemplate.systemPrompt удалено: системный промт этапа = только roleContext

Status: active

Decision:

Поле `StageTemplate.systemPrompt` («Дополнение к системному промту этапа», редактировалось в `TemplateEditorView`) удалено из модели и всей связанной логики (FT-20260704-001). `PromptBuilder.build(...)` больше не склеивает `roleContext` с `template.systemPrompt` — итоговый `system`-промт равен просто `roleContext` (роль + редполитика/источники из `AIRole`/`ContextBlock`). Перед удалением проверена реальная база приложения (`default.store`) — поле было пустым во всех 8 сохранённых шаблонах на момент удаления, потери данных не было. Явного шага миграции SwiftData не потребовалось: в проекте нет `VersionedSchema`/`SchemaMigrationPlan`, схема выводится автоматически, а удаление одного строкового атрибута — стандартный случай lightweight-миграции.

Why:

Пользователь усомнился, используется ли поле вообще — проверка показала, что оно было реально подключено в промт-пайплайн, но ни один из 9 стандартных шаблонов его не заполнял, и пользователь ни разу не вписал туда текст вручную. Отдельный уровень «системный промт конкретного этапа» поверх роли/редполитики оказался невостребованным на практике — решили не поддерживать его дальше, а не оставлять неиспользуемую настройку.

Impact:

Если в будущем понадобится точечно переопределять системный промт для одного этапа (а не через роль/редполитику, которые общие для всех этапов роли), это будет новая функциональность, а не восстановление старого поля — старое поле обсуждать бессмысленно, код удалён полностью (модель, дефолты, сидер, `PromptBuilder`, `StageExecutor.fetchStageTemplatesSummary`, `TemplateEditorView` и резервные конструкторы в `TopicWorkspaceView`/`QuickCheckSheet`/`StructureEditorSheet`).

### 2026-07-03 — Новая роль в RoleDefaults требует явного шага миграции, не только сидинга

Status: active

Decision:

`StageTemplateSeeder.seedRolesIfNeeded` заполняет `AIRole` только при полностью пустой таблице (`guard existing.isEmpty else return`). Поэтому при добавлении новой записи в `RoleDefaults.all` (см. роль `"analyst"` для FT-20260703-003) для уже существующих инсталляций её нужно отдельно вставить в `migrateTemplatesIfNeeded` (проверка `!roles.isEmpty && !roles.contains(where: { $0.key == новый_key })`) и обязательно поднять `StageTemplateSeeder.currentTemplatesDefaultsVersion`. Аналогично, добавление нового `PipelineStage` (не `.action`) авто-досевается в `StageTemplate` благодаря `seedStageTemplatesIfNeeded`, которая проверяет каждый этап по отдельности, а не общую пустоту таблицы — для шаблонов отдельного шага миграции не требуется, только для ролей/контекстных блоков/скиллов, которые сидятся по принципу «таблица пуста».

Why:

При реализации этапа «Анализ и обучение» обнаружилось, что просто добавить роль в `RoleDefaults.all` недостаточно — на уже установленных базах данных `seedRolesIfNeeded` никогда не перезапустится (таблица ролей не пуста), и агент этапа остался бы без нужного мандата/имени, откатываясь на дефолт `stage.agentName`. Явный шаг миграции с bump версии — единственный надёжный способ доставить новую роль в существующие инсталляции, по аналогии с уже существующей миграцией `SkillPreset` "shorten".

Impact:

Любая будущая новая роль в `RoleDefaults.all` должна сопровождаться и bump'ом `currentTemplatesDefaultsVersion`, и точечной вставкой в `migrateTemplatesIfNeeded` — иначе она не появится у пользователей с уже существующей базой. Стадии (`StageTemplate`) этого не требуют — там достаточно просто добавить `PipelineStage`-кейс.

### 2026-07-04 — PromptRecommendationsSheet получил «Применить», но без автосохранения

Status: active

Decision:

Решение 2026-07-03 ниже описывало `PromptRecommendationsSheet` как «только для чтения, без автоприменения». Добавлена кнопка «Применить» (`PromptFixApplySheet.swift`), которая открывает редактируемый черновик: пользовательский промт выбранного этапа + вставленная рекомендация, с diff-подсветкой (переиспользует `ParagraphDiff`). Запись в `StageTemplate.userPromptTemplate` происходит только по явному нажатию «Сохранить в шаблон» — автоматического применения по-прежнему нет. Целевой этап для правки подбирается эвристикой (сопоставление `PromptRecommendation.location` с `PipelineStage.title`), пользователь может сменить этап в Picker'е перед сохранением.

Why:

Пользователь явно попросил возможность применить рекомендацию, но настоял на праве редактировать итоговый промт перед сохранением и видеть разницу «было/стало» — то есть исходный принцип «пользователь решает сам» сохраняется, автоматизируется только рутина (открыть нужный шаблон, вставить текст).

Impact:

`recommendation.location` остаётся свободным текстом от ИИ, а не структурированной ссылкой на `StageTemplate` — при добавлении новых источников рекомендаций эвристику подбора этапа нужно будет пересмотреть, если она перестанет справляться. Поле «Дополнение к системному промту этапа» (`StageTemplate.systemPrompt`), упомянутое здесь как не использованное для применения правок, удалено целиком 2026-07-04 — см. запись ниже.

### 2026-07-03 — PipelineStage.promptAnalysis: обычный чат-этап со StageKind.analysis, результат не версия и не remarks

Status: active

Decision:

`StageKind` получил третий содержательный кейс `.analysis` (кроме `.author`/`.checking`/`.action`). В отличие от `.action` (`.images`), этап `.promptAnalysis` — обычный чат-completion этап: у него есть `StageTemplate`, он идёт через `StageExecutor.execute()` тем же путём, что и остальные этапы (`TopicWorkspaceView.runSelectedStage()` не перехватывает его отдельно). Но результат — не `ArticleVersion` и не `executor.remarks`, а новая сущность `PromptRecommendation` (cascade от `Topic` и от `GenerationJob`), показывается только для чтения в `PromptRecommendationsSheet`, без автоприменения.

Why:

Результат этапа — рекомендации по правке промтов других этапов, а не правка текста статьи, поэтому ни существующий механизм `ArticleVersion` (принятие/отклонение версии), ни `Remark`/`RemarkApplier` (замена цитаты в тексте) не подходят по смыслу. Impact:

Любой новый код с exhaustive `switch` над `StageKind` (уже задело `StageExecutor.execute()`, `StageProgress.isCompleted`, `StageRunGuard`, `StageVersionsSummaryBuilder`, `StageExecutor.fetchStageTemplatesSummary`) должен явно решить, что делать с `.analysis`. Если в будущем появится ещё один этап с результатом-не-версией (например, FT-20260703-005 «Правки по комментариям врачей», если решат делать отдельным этапом) — стоит сверяться с этим паттерном или с паттерном `.action` (`.images`) в зависимости от того, нужен ли чат-вызов вообще.

`PipelineStage` получил кейс `.images` (генерация изображений) и `StageKind` — кейс `.action`. Любой новый код, добавляющий `case` в exhaustive `switch` над `PipelineStage` (уже задело `StageTemplateDefaults.content(for:)`) или над `StageKind`, должен явно решить, что делать с `.images`/`.action`. Status: active

Ручной редакционный каскад из 8 промтов хранится как дефолтные `ContextBlock`, `Role`, `StageTemplate` и `SkillPreset`, а не как live-импорт из Google Docs/Sheets. Эта запись остаётся как история устройства каскада версий 3–8; для промтов начиная с v9 действует новое правило.

Why:

Так приложение сохраняет утверждённый ручной процесс внутри повседневного запуска, оставляет тексты редактируемыми для пользователя и не зависит от доступности внешних Google-файлов во время генерации.

Impact:

Новые SwiftData-контейнеры должны регистрировать `ForbiddenPhrase`. Новые промты, использующие список запретов, должны получать данные через `ForbiddenPhraseRenderer`; пустой список рендерится как `(список пуст)`. Будущие миграции дефолтов должны оставаться версионированными и узкими, чтобы не перезаписывать пользовательские шаблоны вне согласованного каскада.

### 2026-07-02 — Публикуемый Google Doc получает H1, но не Title/Description

Status: active

Decision:

При публикации (`ArticlePublisher.normalizeHeading`) заголовок H1 в теле документа синхронизируется с `ArticleVersion.h1` (SEO-утверждённым заголовком из этапа «Семантика-в-текст»), если он задан — заменяет или добавляет строку `# {h1}` в начале текста. `seoTitle`/`seoDescription` в тело документа НЕ публикуются.

Why:

H1 — часть читаемого контента статьи, поэтому документ должен показывать именно тот заголовок, который прошёл SEO-проверку, а не случайный заголовок черновика. Title/Description — метаданные для HTML `<title>`/`<meta description>` сайта, а не для читателя Google Doc; вставка их в тело документа создавала бы путаницу для редактора. Решение принято автономно (агент работал без пользователя, FT-20260702-007) — при возражении пользователя нужно пересмотреть.

Impact:

Если в будущем появится потребность публиковать Title/Description отдельно (например, для передачи в CMS), делать это нужно отдельным полем/каналом, а не смешивать с телом Google Doc.

### 2026-07-01 — Перезапись Google Docs должна быть атомарной

Status: active

Decision:

При перезаписи опубликованного Google Docs нельзя сначала очищать документ отдельным запросом, а затем вставлять новый текст отдельным запросом. Удаление старого содержимого и вставка нового должны уходить одним batch-запросом. Пользователь должен выбирать конкретную прошлую публикацию для перезаписи, если у темы их несколько.

Why:

Если очистка и вставка разделены, сбой между ними может оставить документ пустым. Один batch-запрос снижает риск частичной перезаписи, а явный выбор документа защищает от случайной перезаписи не того опубликованного материала.

Impact:

Новые изменения в публикации должны сохранять это правило. Live-проверки лучше делать на одноразовом тестовом Google Doc.

### 2026-07-02 — Семантика темы: решения по ключевым словам вместо списка строк

Status: active

Decision:

Семантика темы хранится как записи `SemanticKeyword` (текст, частотность, рекомендация агента, решение пользователя: pending/accepted/rejected/required, причина, риск каннибализации) вместо простого списка строк `Topic.semantics`. Старое поле `Topic.semantics` не удаляется и продолжает работать как fallback: `SemanticPromptRenderer` берёт `{{семантика}}` из принятых/обязательных `SemanticKeyword`, а если таких записей ещё нет — использует старые строки. Индекс опубликованных страниц сайта (`PublishedSitePage`) обновляется только вручную по кнопке, автоматического фонового обновления нет; при сбое обновления старый индекс не удаляется.

Why:

Пользователь должен сам решать, какие запросы включать в семантику, а не доверять автоматике безоговорочно — агент только предлагает (include/exclude) с объяснением и риском каннибализации. Сохранение legacy-поля и данных при сбоях защищает существующие темы от потери семантики при миграции.

Impact:

Новый код, читающий семантику темы для промптов, должен использовать `SemanticPromptRenderer`, а не `topic.semantics` напрямую. Любые новые SwiftData-модели должны быть зарегистрированы и в `SEOContentCreatorApp.swift`, и во всех тестовых `ModelContainer`, использующих `Topic.self`.

### 2026-07-01 — Новая версия статьи сначала pending, затем явно accepted/rejected

Status: active

Decision:

Свежесгенерированная `ArticleVersion` должна иметь явное состояние ожидания решения пользователя (`pending`) и становиться текущей принятой версией только после явного принятия. Отклонение должно сохранять понятный статус, а не выглядеть как обычная текущая версия.

Why:

Главный инвариант продукта — человек принимает результат. Неявно считать свежую генерацию принятой опасно: пользователь может перепутать черновой результат с утверждённым текстом.

Impact:

Новые генераторы и правки текста должны уважать поток `pending` → `accepted` / `rejected` и не обходить ручное принятие.

### 2026-06-29 — Текстовая модель выбирается только в Настройках

Status: superseded

Decision:

Для текстовой генерации пользователь выбирает модель только через `@AppStorage("openAIModel")` в `SettingsView`. Редактор промтов этапов не содержит отдельный Picker модели и не сохраняет `StageTemplate.modelName` при правке шаблона. Runtime-запуски (`execute`, `executeQuickCheck`, `executeSandbox`) должны получать модель из настроек через параметр override. Старое поле `StageTemplate.modelName` остаётся в модели данных как fallback для совместимости со старыми шаблонами и тестами.

Update 2026-07-09: решение заменено новым правилом «Модель и reasoning задаются на уровне StageTemplate». `StageTemplate.modelName` снова является активной пользовательской настройкой для пайплайновых этапов, а runtime-запуски этапа/песочницы шаблона не должны подменять её глобальной моделью.

Два места выбора модели создавали неочевидное поведение: пользователь менял модель в Настройках, но конкретный шаблон мог продолжать запускаться на своей сохранённой модели. Один источник выбора проще для редактора и снижает риск случайно прогонять разные этапы на разных моделях.

Impact:

Новые UI-пути запуска текста должны передавать модель из `openAIModel`, а не читать `template.modelName` напрямую. Удалять `StageTemplate.modelName` без отдельной миграции нельзя: поле нужно как обратная совместимость и fallback.

### 2026-06-27 — executeQuickCheck: транзитный Topic не вставляется в контекст

Status: active

Decision:

В `StageExecutor.executeQuickCheck` объект `Topic(title: "", articleType: .info)` создаётся транзитно — только как аргумент `PromptBuilder.build(template:topic:currentText:...)` — и **не** передаётся в `context.insert(...)`. SwiftData не отслеживает не вставленные объекты. Никаких GenerationJob, ArticleVersion или Topic в базе не появляется.

Why:

«Быстрая проверка» работает с произвольным текстом без привязки к теме; любой `context.insert(scratch)` создал бы «призрачную» тему. Транзитный объект — минимальная совместимость с сигнатурой PromptBuilder без изменения его интерфейса.

Impact:

Нельзя добавлять `context.insert(scratch)` в `executeQuickCheck` — это нарушит инвариант «быстрая проверка ничего не сохраняет». Если PromptBuilder в будущем потребует `persistentModelID` — потребуется рефакторинг интерфейса.

### 2026-07-03 — Ссылки Topic → KnowledgeNode обязаны иметь `inverse` и `deleteRule: .nullify`

Status: active

Decision:

Любая связь `@Relationship` из `Topic` на `KnowledgeNode`, хранящая FK-колонку прямо в таблице `Topic` (`direction`, `doctor` — не `attachedNodes`, у него FK на стороне ребёнка), обязана объявлять и `inverse:` на соответствующее свойство в `KnowledgeNode`, и `deleteRule: .nullify`. Обратные свойства — `KnowledgeNode.topicsUsingAsDirection`/`topicsUsingAsDoctor`.

Why:

Без `inverse` SwiftData физически не может найти темы, ссылающиеся на удаляемый узел, поэтому `deleteRule` без `inverse` — это no-op (проверено регрессионным тестом `TopicKnowledgeNodeDeletionTests`, который сначала падал с одним лишь `deleteRule`, без `inverse`). Баг был найден на живых данных пользователя: удаление направления «Урология» в Базе знаний оставило тему с висящей ссылкой (`ZDIRECTION` указывал на несуществующую строку), и приложение падало на каждом запуске при попытке `SwiftData` прочитать `direction?.title` в `ContentPlanView`. См. `ai/changelog.md` за эту дату.

Impact:

Аддитивное изменение схемы (два новых relationship-массива в `KnowledgeNode`, инициализируются пустыми) — существующие данные не мигрируют деструктивно. Любая новая to-one связь `Topic → KnowledgeNode` в будущем должна следовать этому же паттерну, иначе повторится тот же класс краша.

### 2026-06-26 — Сопоставление с заводскими дефолтами по стабильному `defaultKey`, не по имени

Status: active

Decision:

Продуктовые блоки (`ProductBlock`) и скиллы (`SkillPreset`) связаны со своими заводскими дефолтами через опциональное поле `defaultKey: String?`, а не через совпадение имени. У каждого дефолта в `ProductBlockDefaults`/`SkillPresetDefaults` есть стабильный `key`; `make(...)` проставляет его в `defaultKey` при сидинге. Сидеры при непустой таблице делают одноразовый `backfillDefaultKeys(...)` — записям с `defaultKey == nil` проставляют ключ по совпадению имени. Редакторы в `TemplatesView` ищут дефолт по `defaultKey`, с откатом на имя только если ключ отсутствует.

Why:

Имя редактируется пользователем; привязка «сбросить к стандартному» к имени ломалась при переименовании заводского блока/скилла (кнопка исчезала). Стабильный ключ не зависит от имени. Замещает часть решения 2026-06-25 («сброс работает по совпадению имени с `SkillPresetDefaults`») — теперь сопоставление по ключу.

Impact:

Аддитивное опциональное поле SwiftData — лёгкая миграция, существующие данные не теряются. Записи, переименованные ДО появления `defaultKey`, ключ через backfill не получат (backfill сопоставляет по имени) — для них работает откат на имя. Новые `key` в дефолтах нельзя менять задним числом: они — стабильный идентификатор связи с засеянными/забэкфилленными записями.

### 2026-06-26 — Продуктовые блоки: @Model ProductBlock + вставка в промт через плейсхолдер с fallback

Status: active

Decision:

Продуктовые блоки хранятся в отдельной аддитивной `@Model ProductBlock` (uuid/name/prompt/order), не встроены в `StageTemplate`. Стартовые 4 блока создаёт `ProductBlockSeeder.seedIfNeeded(in:)` при запуске (идемпотентно, по пустоте таблицы), правятся в разделе «Продуктовые блоки» в «Шаблонах» (по образцу «Скиллы»). При генерации стадии `productBlocks` промт каждого выбранного блока проходит ту же подстановку KB-переменных, что и шаблон стадии (`PromptBuilder.substitute`), и собранные промты вставляются в шаблон стадии через плейсхолдер `{{продуктовые_блоки}}`. Если плейсхолдера в шаблоне нет — промты дописываются в конец (fallback). `ProductBlocksSheet` отдаёт промты блоков (не имена) через неизменную сигнатуру `onGenerate: ([String]) -> Void`.

Why:

Продуктовые блоки семантически независимы от этапов пайплайна и должны редактироваться отдельно. Плейсхолдер `{{продуктовые_блоки}}` даёт явный контроль места вставки в новых/сброшенных шаблонах, а fallback гарантирует совместимость со старыми сохранёнными шаблонами стадии, у которых плейсхолдера нет. Данные врача/преимуществ берутся из Базы знаний (через переменные внутри промта блока), а не выдумываются ИИ.

Impact:

Любой `ModelContainer` (включая тестовые) должен регистрировать `ProductBlock` — иначе runtime-ошибка. Добавление в схему — лёгкая миграция SwiftData (новая сущность, без явного MigrationPlan). Fallback-ветку в `PromptBuilder` ломать нельзя — на ней держится совместимость старых шаблонов стадии `productBlocks`.

### 2026-06-25 — SkillPreset: отдельная @Model сущность, стандартные скиллы через Seeder

Status: active

Decision:

Пресеты скиллов хранятся в `@Model SkillPreset` (uuid/name/prompt/roleKey/order) — отдельная SwiftData-сущность, не встроенная в `StageTemplate`. Стартовые 4 скилла создаются через `SkillPresetSeeder.seedIfNeeded(in:)` при запуске (как `EditorDictionarySeeder`), идемпотентно — проверяет по count > 0. Пресеты редактируются в разделе «Скиллы» в «Шаблонах» (load/save/reset/delete), сброс к стандартному работает по совпадению имени с `SkillPresetDefaults`. Замена фрагмента по строке — через `FragmentSplicer` с проверкой уникальности: ровно 1 совпадение → `.replaced`, иначе `.notFound`/`.ambiguous(N)`.

Why:

Пресеты скиллов семантически независимы от этапов пайплайна и не должны лежать в `StageTemplate`. Отдельная модель даёт отдельный CRUD, отдельный раздел в «Шаблонах» и изолированные тесты. Seeder-паттерн уже есть (`EditorDictionarySeeder`) — повторное использование снижает когнитивную нагрузку.

Impact:

Любой `ModelContainer` (включая тестовые) должен регистрировать `SkillPreset` — иначе runtime-ошибка. Добавление `SkillPreset` в схему — лёгкая миграция SwiftData (новая сущность, без явного MigrationPlan; существующие данные не затрагиваются). `FragmentEditor` использует тот же `StreamProvider`, что и `StageExecutor` — типовой псевдоним, не наследование.

### 2026-06-24 — Мягкие подсказки правки: алгоритмические, не через ИИ, не блокируют

Status: active

Decision:

Подсказки правки (длинные предложения, повторы однокоренных рядом, штампы) считаются чисто алгоритмически в `Logic/SoftHints.swift` — без вызова ИИ. Они показываются в отдельном `SoftHintsSheet` поверх текущей версии, ничего не сохраняют, не создают `ArticleVersion` и не блокируют сохранение/работу. Пороги и словарь штампов хранятся в одной seed-записи `@Model EditorDictionary` и правятся в «Шаблонах» (раздел «Словарь правок») со сбросом к стандартному. Это сознательно отделено от проверяющего этапа «Финальная вычитка» (который идёт через ИИ-редактора и создаёт замечания для точечного приёма).

Why:

Грубые механические проблемы (длина фразы, тавтология, канцелярит-штампы) дешевле и мгновеннее ловить правилом, чем запросом к модели — без расхода токенов и ожидания. Отделение от «Финальной вычитки» исключает дублирование и сохраняет инвариант «человек принимает»: подсказки лишь подсвечивают, решение и правка остаются за редактором.

Impact:

Любой `ModelContainer`, регистрирующий схему, должен также регистрировать `EditorDictionary` — иначе контейнер падает в runtime (тестовые контейнеры обновлены). Сидер (`EditorDictionarySeeder`, вызов в `RootView`) создаёт запись при первом запуске; добавление новой сущности — лёгкая миграция SwiftData без явного MigrationPlan. Однокоренной детектор — грубый префиксный (приемлемо по спеке); при необходимости точности его можно заменить, не меняя контракт sheet. Пункты 1 (пресеты скиллов) и 2 (регенерация фрагмента) того же FT остаются в `ai/future-tasks.md`.

### 2026-06-24 — reasoning_effort: поле шаблона, отправка только для новых моделей

Status: active

Decision:

«Интенсивность мышления» (`reasoning_effort`) хранится в опциональном поле `StageTemplate.reasoningEffort: String?` ("low"/"medium"/"high"; `nil` = не передавать параметр). В тело запроса OpenAI параметр `reasoning_effort` добавляется только когда модель попадает под `OpenAIClient.usesMaxCompletionTokens(model:)` (GPT-5.x / o1 / o3 / o4 / chat-latest) И значение не `nil`. Для legacy-моделей (gpt-4.x и т.п.) параметр не отправляется никогда. Имя модели не хардкодится — гейтинг всегда через `usesMaxCompletionTokens`.

Why:

`reasoning_effort` поддерживают только рассуждающие модели (GPT-5/o-серия); отправка legacy-моделям вызвала бы ошибку API. Привязка гейтинга к уже существующей функции `usesMaxCompletionTokens` (которая уже различает эти семейства для `max_completion_tokens`) исключает дублирование списка моделей и расхождения. Опциональное поле с `nil`-по-умолчанию даёт лёгкую миграцию и обратную совместимость старых шаблонов.

Impact:

Любой код, вызывающий `OpenAIClient.streamCompletion`, может передать `reasoningEffort`. `StageExecutor.StreamProvider` теперь 7-аргументный (добавлен `reasoningEffort: String?`) — все провайдеры (в т.ч. тестовые) должны соответствовать. Добавление поля в `@Model StageTemplate` — лёгкая миграция SwiftData (контейнер без явного MigrationPlan; старые шаблоны получают `nil`). UI-гейтинг Picker'а в `TemplatesView` использует ту же `usesMaxCompletionTokens`.

### 2026-06-20 — Отказ от очереди и фоновой automation

Status: active

Decision:

Очередь и фоновая последовательная automation убраны из продукта **насовсем**. Удалены: раздел «Очередь» в интерфейсе, сущности «Очередь» (Queue) и «Сценарий запуска» (Run Scenario), виджет очереди в сайдбаре. В спеке (`docs/superpowers/specs/2026-05-19-content-system-redesign-design.md`, версия 9) убраны раздел 8, подраздел 4.3, сущности 2.9 и 2.11 и все ссылки на очередь в Контент-плане и рабочем пространстве. Основных разделов приложения стало 4 (Контент-план, Рабочее пространство темы, Шаблоны, База знаний). Темы обрабатываются по одной, вручную, этап за этапом.

Why:

Фоновая прогонка цепочки этапов конфликтует с базовым инвариантом «человек принимает». В коде каждый генерирующий этап (`StageExecutor`) создаёт версию, которую редактор принимает вручную в side-by-side, а проверяющие этапы выдают замечания для точечного приёма. Чтобы сцепить этапы в фоне, пришлось бы авто-принимать результаты без участия человека — это размывает контроль качества, ради которого и строился продукт. Для single-user сценария ценность фоновой очереди не оправдывает усложнение и риск.

Impact:

Удалены реальные ветки кода: пункт `.queue` в `SidebarView` и заглушка раздела очереди в `RootView`. Сущностей/моделей данных очереди в коде не было (не реализованы) — миграции не требуются. Номера удалённых блоков в спеке (2.9, 2.11, 4.3, раздел 8) намеренно оставлены пропусками, чтобы сохранить перекрёстные ссылки. Исторические планы в `docs/superpowers/plans/` не переписываются. Инвариант «нет авто-публикации» сохранён и перенесён в раздел 9 спеки.

### 2026-06-20 — Отказ от опционального этапа «Стиль/Главред»

Status: active

Decision:

Опциональный этап «Стиль/Главред» убран из продукта. Он создавался под подключение внешнего сервиса «Главред», от которого решено отказаться. Из кода удалены поле `Topic.useStyle` и галочка «Использовать Стиль/Главред» в `BriefView`. Самого этапа в `PipelineStage` никогда не было — существовала только эта заготовка-флаг. Стиль и язык текста закрывает обязательный этап «Финальная вычитка» (агент ИИ-редактор). Видимых этапов теперь 9 обязательных + 1 опциональный (Продуктовые блоки); раньше было 9 + 2.

Why:

Этап имел смысл только как обёртка над внешним сервисом «Главред». Без него отдельная стилевая правка дублирует «Финальную вычитку» (та же роль ИИ-редактора, та же зона — язык и стиль). Один этап вместо двух проще для редактора и убирает мёртвую заготовку из модели данных и интерфейса.

Impact:

Спека обновлена до версии 8 (`docs/superpowers/specs/2026-05-19-content-system-redesign-design.md`), затронуты также frontend-design и checking-stages-design спеки. Удаление свойства `useStyle` из `@Model Topic` — лёгкая миграция SwiftData (поле просто исчезает у существующих тем). Исторические планы (`docs/superpowers/plans/*`) намеренно не переписаны — это записи уже выполненных под-проектов.



### 2026-06-23 — OAuth loopback: POSIX BSD-socket вместо NWListener

Status: active

Decision:

Для локального OAuth loopback-сервера (перехват redirect от Google) использовать обычный POSIX BSD-сокет (`socket/bind/listen/accept`) вместо `NWListener` из фреймворка Network. Привязка строго к `127.0.0.1`.

Why:

`NWListener` падает с EINVAL (POSIX 22) на macOS 26.x при любой конфигурации (`.any`, `.loopback`, явный порт, эфемерный порт). Корень проблемы в самом фреймворке, не в коде. POSIX-сокет работает штатно и даёт ту же loopback-only гарантию через `sin_addr = inet_addr("127.0.0.1")`.

Impact:

Файл `GoogleAuthService.swift` (класс `LoopbackListener`). Зависимость `import Network` убрана. Если на будущей macOS `NWListener` починят — можно вернуть, но тест будет не CLI-автоматизируемым (интерактивный браузер). Решение актуально для любых компонентов проекта, которым нужен локальный TCP-слушатель.

### 2026-06-20 — Галерея изображений отдельно от ленты версий; render-then-save split

Status: active

Decision:

Изображения (`GeneratedImage`) хранятся в отдельной галерее на `Topic` (связь `@Relationship(deleteRule: .cascade)`), не смешиваясь с `ArticleVersion`. `Topic.coverImageID` — UUID, указывающий на текущую обложку. Данные изображения хранятся с `@Attribute(.externalStorage)` (большие бинарные данные). Иллюстрация сохраняет `anchorQuote` (якорь-цитата) и `sourceImageID` (ссылка на исходник при доработке). Доработка только промтом: новая картинка создаётся как отдельная запись с `sourceImageID`, исходник не изменяется. Логика разделена на два независимых класса: `ImageGenerator` (рендер в `previewData`, лог `GenerationJob`) и `ImageSaver` (persist в галерею только по явному действию пользователя «Сохранить»).

Why:

Текст статьи и изображения — разные артефакты с разным жизненным циклом. Объединение в одну ленту загромоздило бы её и усложнило отображение. Разделение `ImageGenerator`/`ImageSaver` повторяет инвариант «человек принимает» и делает сетевой путь и путь сохранения тестируемыми независимо. `@externalStorage` обязателен для поля `data` — без него SwiftData хранит blob в основной БД, что замедляет запросы при большом числе изображений.

Impact:

Любой `ModelContainer`, регистрирующий `Topic`, должен также регистрировать `GeneratedImage`, `ImageStylePreset`, `ImagePromptTemplate` — иначе контейнер падает в runtime. Тестовые контейнеры в unit-тестах обновлены соответственно. Вставка изображений в Google Docs — будущий этап «Публикация» (не в объёме под-проекта 8); якорь-цитата сохранена как задел.

### 2026-05-24 — Этап «Структура» отдельно от текста; план как поле темы

Status: active

Decision:

ИИ-автор сначала строит план статьи, потом пишет текст. Добавлен отдельный первый этап пайплайна `PipelineStage.structure` (роль `author`): он генерирует план H1/H2/H3 с краткими пометками по разделам, без ключевых запросов. План хранится в `Topic.structureText` — редактируемом поле темы (по образцу `Topic.semantics`), а не как версия в ленте. Этап «Черновик» пишет текст по плану через новую переменную `{{структура}}`.

Why:

Раньше первый этап «Черновик» делал структуру и текст за один проход. Для длинных медицинских статей раздельный подход даёт лучший контроль (план — это SEO-каркас), дешёвую правку до генерации полного текста и человеческую контрольную точку, что согласуется с инвариантом «человек принимает». Хранение планом-полем выбрано вместо версии в ленте, потому что ручного редактирования версий в приложении пока нет, а план — рабочий вход, а не публикуемый текст; поле темы сразу редактируемо и повторяет готовый паттерн «Семантики».

Impact:

`StageTemplate` для `.draft` переписан на использование `{{структура}}` с мягким фолбэком (пустой план — пишем с нуля). Генерация структуры не создаёт `ArticleVersion`. Новый этап досевается автоматически (сидер добавляет недостающие этапы), миграция дефолтов не нужна. Изменение модели `Topic` — лёгкая миграция SwiftData (у старых тем `structureText` пустой). Семантика остаётся отдельным поздним этапом без ключей в структуре.

### 2026-05-23 — ИИ-роли и блоки контекста как редактируемые сущности

Status: active

Decision:

Роль ИИ-агента и общие методички хранятся отдельно от промтов этапов. В SwiftData добавлены `AIRole` (фиксированные ключи `author`, `seo`, `factChecker`, `editor`) и `ContextBlock` (фиксированные ключи `editorialPolicy`, `sources`, `seoGuidelines`). Этапы пайплайна имеют фиксированный `PipelineStage.roleKey`, а полный system prompt собирается как: мандат роли → тексты подключённых блоков в каноническом порядке → короткое дополнение этапа.

Why:

Раньше роль, редполитика, источники и SEO-правила дублировались в системных промтах этапов. Это усложняло правку и повышало риск расхождений. Отдельные роли и блоки дают правку в одном месте, но не расширяют скоуп до произвольного создания ролей или переназначения этапов.

Impact:

`StageTemplate.systemPrompt` теперь должен содержать только этап-специфичное дополнение, часто пустое. Пользовательский промт этапа, параметры модели и история запусков остаются отдельными. Новые запуски сохраняют имя роли как снимок в `GenerationJob.agentName` и `ArticleVersion.agentName`; старую историю не мигрируем.

### 2026-05-21 — Стек приложения: нативный SwiftUI

Status: active

Decision:

Приложение реализуется как нативное macOS-приложение на SwiftUI. Веб-обёртка (Tauri/Electron) отклонена.

Why:

Нужен только macOS — кроссплатформенность веб-обёртки не требуется. Один цельный стек (Swift + SwiftUI) проще вести не-разработчику с ИИ-сопровождением и надёжнее, чем связка «веб-фронт + упаковщик». Нативный вид и ощущение, хорошая поддержка ИИ. Целевые экраны (side-by-side, дерево Базы знаний, списки, формы) хорошо ложатся на SwiftUI.

Impact:

Skill frontend-design (веб-ориентированный) напрямую не применяется — UI проектируется через brainstorming + макеты и реализуется в SwiftUI. Потребуется Xcode и разовая настройка подписи приложения. Следующий шаг — фронтенд-дизайн ключевых экранов.

### 2026-05-21 — Продуктовая архитектура нового приложения (Вариант B)

Status: active

Decision:

Принят Вариант B (смелое переосмысление) для редизайна приложения. Ключевое: единая лента версий вместо слотов; side-by-side как основной экран; короткий видимый пайплайн (8 обязательных + 2 опциональных этапа), технические этапы скрыты в контекст ИИ-ролей; ИИ-роли (автор, SEO, фактчекер, редактор) с независимой проверкой; База знаний клиники (древовидный справочник); структурированный бриф (направление/врач из Базы знаний) вместо тегов; Google Docs только как точка публикации без обратной синхронизации; ручная правка сквозная.

Why:

Главная боль старого приложения — слишком много видимых этапов и непрозрачность артефактов. Вариант B убирает это, сохраняя полезную суть. Single-user снимает переходный барьер.

Полная спека — `docs/superpowers/specs/2026-05-19-content-system-redesign-design.md`. Перед реализацией нужно выбрать стек (нативный macOS / веб). Отложено: автоимпорт из Docs, аналитика конкурентов.

### 2026-05-17 — Plan files are source of truth for plan-driven progress

Status: active

Decision:

For Superpowers or other plan-driven workflows, `docs/superpowers/plans/<plan>.md` is the source of truth for execution progress. Agents must update plan checkboxes after each completed plan task, add short `Note:` entries for local judgment calls, and use `Plan Task <N>: <short action>` commit messages for plan-driven commits.

Why:

Internal TodoWrite state is session-local and disappears when switching agents or chats. Keeping progress in the plan file and using a simple commit convention makes handoff verifiable without extra tools.

Impact:

Ordinary non-plan-driven tasks are unchanged. `ai/decisions.md` remains for durable decisions, `ai/changelog.md` remains for notable summaries, and plan notes are used only for local decisions inside a specific plan.
