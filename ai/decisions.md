## Текущие решения

### 2026-07-01 — Перезапись Google Docs должна быть атомарной

Status: active

Decision:

При перезаписи опубликованного Google Docs нельзя сначала очищать документ отдельным запросом, а затем вставлять новый текст отдельным запросом. Удаление старого содержимого и вставка нового должны уходить одним batch-запросом. Пользователь должен выбирать конкретную прошлую публикацию для перезаписи, если у темы их несколько.

Why:

Если очистка и вставка разделены, сбой между ними может оставить документ пустым. Один batch-запрос снижает риск частичной перезаписи, а явный выбор документа защищает от случайной перезаписи не того опубликованного материала.

Impact:

Новые изменения в публикации должны сохранять это правило. Live-проверки лучше делать на одноразовом тестовом Google Doc.

### 2026-07-01 — Новая версия статьи сначала pending, затем явно accepted/rejected

Status: active

Decision:

Свежесгенерированная `ArticleVersion` должна иметь явное состояние ожидания решения пользователя (`pending`) и становиться текущей принятой версией только после явного принятия. Отклонение должно сохранять понятный статус, а не выглядеть как обычная текущая версия.

Why:

Главный инвариант продукта — человек принимает результат. Неявно считать свежую генерацию принятой опасно: пользователь может перепутать черновой результат с утверждённым текстом.

Impact:

Новые генераторы и правки текста должны уважать поток `pending` → `accepted` / `rejected` и не обходить ручное принятие.

### 2026-06-29 — Текстовая модель выбирается только в Настройках

Status: active

Decision:

Для текстовой генерации пользователь выбирает модель только через `@AppStorage("openAIModel")` в `SettingsView`. Редактор промтов этапов не содержит отдельный Picker модели и не сохраняет `StageTemplate.modelName` при правке шаблона. Runtime-запуски (`execute`, `executeQuickCheck`, `executeSandbox`) должны получать модель из настроек через параметр override. Старое поле `StageTemplate.modelName` остаётся в модели данных как fallback для совместимости со старыми шаблонами и тестами.

Why:

Два места выбора модели создавали неочевидное поведение: пользователь менял модель в Настройках, но конкретный шаблон мог продолжать запускаться на своей сохранённой модели. Один источник выбора проще для редактора и снижает риск случайно прогонять разные этапы на разных моделях.

Impact:

Новые UI-пути запуска текста должны передавать модель из `openAIModel`, а не читать `template.modelName` напрямую. Удалять `StageTemplate.modelName` без отдельной миграции нельзя: поле нужно как обратная совместимость и fallback.

### 2026-06-27 — executeQuickCheck: транзитный Topic не вставляется в контекст

Status: active

Decision:

В `StageExecutor.executeQuickCheck` объект `Topic(title: "", articleType: .info)` создаётся транзитно — только как аргумент `PromptBuilder.build(template:topic:currentText:...)` — и **не** передаётся в `context.insert(...)`. SwiftData не отслеживает не вставленные объекты. Никаких GenerationJob, ArticleVersion или Topic в базе не появляется.

Why:

«Быстрая проверка» работает с произвольным текстом без привязки к теме; любой `context.insert(scratch)` создал бы «призрачную» тему. Транзитный объект — минимальная совместимость с сигнатурой PromptBuilder без изменения его интерфейса.

Impact:

Нельзя добавлять `context.insert(scratch)` в `executeQuickCheck` — это нарушит инвариант «быстрая проверка ничего не сохраняет». Если PromptBuilder в будущем потребует `persistentModelID` — потребуется рефакторинг интерфейса.

### 2026-06-26 — Сопоставление с заводскими дефолтами по стабильному `defaultKey`, не по имени

Status: active

Decision:

Продуктовые блоки (`ProductBlock`) и скиллы (`SkillPreset`) связаны со своими заводскими дефолтами через опциональное поле `defaultKey: String?`, а не через совпадение имени. У каждого дефолта в `ProductBlockDefaults`/`SkillPresetDefaults` есть стабильный `key`; `make(...)` проставляет его в `defaultKey` при сидинге. Сидеры при непустой таблице делают одноразовый `backfillDefaultKeys(...)` — записям с `defaultKey == nil` проставляют ключ по совпадению имени. Редакторы в `TemplatesView` ищут дефолт по `defaultKey`, с откатом на имя только если ключ отсутствует.

Why:

Имя редактируется пользователем; привязка «сбросить к стандартному» к имени ломалась при переименовании заводского блока/скилла (кнопка исчезала). Стабильный ключ не зависит от имени. Замещает часть решения 2026-06-25 («сброс работает по совпадению имени с `SkillPresetDefaults`») — теперь сопоставление по ключу.

Impact:

Аддитивное опциональное поле SwiftData — лёгкая миграция, существующие данные не теряются. Записи, переименованные ДО появления `defaultKey`, ключ через backfill не получат (backfill сопоставляет по имени) — для них работает откат на имя. Новые `key` в дефолтах нельзя менять задним числом: они — стабильный идентификатор связи с засеянными/забэкфилленными записями.

### 2026-06-26 — Продуктовые блоки: @Model ProductBlock + вставка в промт через плейсхолдер с fallback

Status: active

Decision:

Продуктовые блоки хранятся в отдельной аддитивной `@Model ProductBlock` (uuid/name/prompt/order), не встроены в `StageTemplate`. Стартовые 4 блока создаёт `ProductBlockSeeder.seedIfNeeded(in:)` при запуске (идемпотентно, по пустоте таблицы), правятся в разделе «Продуктовые блоки» в «Шаблонах» (по образцу «Скиллы»). При генерации стадии `productBlocks` промт каждого выбранного блока проходит ту же подстановку KB-переменных, что и шаблон стадии (`PromptBuilder.substitute`), и собранные промты вставляются в шаблон стадии через плейсхолдер `{{продуктовые_блоки}}`. Если плейсхолдера в шаблоне нет — промты дописываются в конец (fallback). `ProductBlocksSheet` отдаёт промты блоков (не имена) через неизменную сигнатуру `onGenerate: ([String]) -> Void`.

Why:

Продуктовые блоки семантически независимы от этапов пайплайна и должны редактироваться отдельно. Плейсхолдер `{{продуктовые_блоки}}` даёт явный контроль места вставки в новых/сброшенных шаблонах, а fallback гарантирует совместимость со старыми сохранёнными шаблонами стадии, у которых плейсхолдера нет. Данные врача/преимуществ берутся из Базы знаний (через переменные внутри промта блока), а не выдумываются ИИ.

Impact:

Любой `ModelContainer` (включая тестовые) должен регистрировать `ProductBlock` — иначе runtime-ошибка. Добавление в схему — лёгкая миграция SwiftData (новая сущность, без явного MigrationPlan). Fallback-ветку в `PromptBuilder` ломать нельзя — на ней держится совместимость старых шаблонов стадии `productBlocks`.

### 2026-06-25 — SkillPreset: отдельная @Model сущность, стандартные скиллы через Seeder

Status: active

Decision:

Пресеты скиллов хранятся в `@Model SkillPreset` (uuid/name/prompt/roleKey/order) — отдельная SwiftData-сущность, не встроенная в `StageTemplate`. Стартовые 4 скилла создаются через `SkillPresetSeeder.seedIfNeeded(in:)` при запуске (как `EditorDictionarySeeder`), идемпотентно — проверяет по count > 0. Пресеты редактируются в разделе «Скиллы» в «Шаблонах» (load/save/reset/delete), сброс к стандартному работает по совпадению имени с `SkillPresetDefaults`. Замена фрагмента по строке — через `FragmentSplicer` с проверкой уникальности: ровно 1 совпадение → `.replaced`, иначе `.notFound`/`.ambiguous(N)`.

Why:

Пресеты скиллов семантически независимы от этапов пайплайна и не должны лежать в `StageTemplate`. Отдельная модель даёт отдельный CRUD, отдельный раздел в «Шаблонах» и изолированные тесты. Seeder-паттерн уже есть (`EditorDictionarySeeder`) — повторное использование снижает когнитивную нагрузку.

Impact:

Любой `ModelContainer` (включая тестовые) должен регистрировать `SkillPreset` — иначе runtime-ошибка. Добавление `SkillPreset` в схему — лёгкая миграция SwiftData (новая сущность, без явного MigrationPlan; существующие данные не затрагиваются). `FragmentEditor` использует тот же `StreamProvider`, что и `StageExecutor` — типовой псевдоним, не наследование.

### 2026-06-24 — Мягкие подсказки правки: алгоритмические, не через ИИ, не блокируют

Status: active

Decision:

Подсказки правки (длинные предложения, повторы однокоренных рядом, штампы) считаются чисто алгоритмически в `Logic/SoftHints.swift` — без вызова ИИ. Они показываются в отдельном `SoftHintsSheet` поверх текущей версии, ничего не сохраняют, не создают `ArticleVersion` и не блокируют сохранение/работу. Пороги и словарь штампов хранятся в одной seed-записи `@Model EditorDictionary` и правятся в «Шаблонах» (раздел «Словарь правок») со сбросом к стандартному. Это сознательно отделено от проверяющего этапа «Финальная вычитка» (который идёт через ИИ-редактора и создаёт замечания для точечного приёма).

Why:

Грубые механические проблемы (длина фразы, тавтология, канцелярит-штампы) дешевле и мгновеннее ловить правилом, чем запросом к модели — без расхода токенов и ожидания. Отделение от «Финальной вычитки» исключает дублирование и сохраняет инвариант «человек принимает»: подсказки лишь подсвечивают, решение и правка остаются за редактором.

Impact:

Любой `ModelContainer`, регистрирующий схему, должен также регистрировать `EditorDictionary` — иначе контейнер падает в runtime (тестовые контейнеры обновлены). Сидер (`EditorDictionarySeeder`, вызов в `RootView`) создаёт запись при первом запуске; добавление новой сущности — лёгкая миграция SwiftData без явного MigrationPlan. Однокоренной детектор — грубый префиксный (приемлемо по спеке); при необходимости точности его можно заменить, не меняя контракт sheet. Пункты 1 (пресеты скиллов) и 2 (регенерация фрагмента) того же FT остаются в `ai/future-tasks.md`.

### 2026-06-24 — reasoning_effort: поле шаблона, отправка только для новых моделей

Status: active

Decision:

«Интенсивность мышления» (`reasoning_effort`) хранится в опциональном поле `StageTemplate.reasoningEffort: String?` ("low"/"medium"/"high"; `nil` = не передавать параметр). В тело запроса OpenAI параметр `reasoning_effort` добавляется только когда модель попадает под `OpenAIClient.usesMaxCompletionTokens(model:)` (GPT-5.x / o1 / o3 / o4 / chat-latest) И значение не `nil`. Для legacy-моделей (gpt-4.x и т.п.) параметр не отправляется никогда. Имя модели не хардкодится — гейтинг всегда через `usesMaxCompletionTokens`.

Why:

`reasoning_effort` поддерживают только рассуждающие модели (GPT-5/o-серия); отправка legacy-моделям вызвала бы ошибку API. Привязка гейтинга к уже существующей функции `usesMaxCompletionTokens` (которая уже различает эти семейства для `max_completion_tokens`) исключает дублирование списка моделей и расхождения. Опциональное поле с `nil`-по-умолчанию даёт лёгкую миграцию и обратную совместимость старых шаблонов.

Impact:

Любой код, вызывающий `OpenAIClient.streamCompletion`, может передать `reasoningEffort`. `StageExecutor.StreamProvider` теперь 7-аргументный (добавлен `reasoningEffort: String?`) — все провайдеры (в т.ч. тестовые) должны соответствовать. Добавление поля в `@Model StageTemplate` — лёгкая миграция SwiftData (контейнер без явного MigrationPlan; старые шаблоны получают `nil`). UI-гейтинг Picker'а в `TemplatesView` использует ту же `usesMaxCompletionTokens`.

### 2026-06-20 — Отказ от очереди и фоновой automation

Status: active

Decision:

Очередь и фоновая последовательная automation убраны из продукта **насовсем**. Удалены: раздел «Очередь» в интерфейсе, сущности «Очередь» (Queue) и «Сценарий запуска» (Run Scenario), виджет очереди в сайдбаре. В спеке (`docs/superpowers/specs/2026-05-19-content-system-redesign-design.md`, версия 9) убраны раздел 8, подраздел 4.3, сущности 2.9 и 2.11 и все ссылки на очередь в Контент-плане и рабочем пространстве. Основных разделов приложения стало 4 (Контент-план, Рабочее пространство темы, Шаблоны, База знаний). Темы обрабатываются по одной, вручную, этап за этапом.

Why:

Фоновая прогонка цепочки этапов конфликтует с базовым инвариантом «человек принимает». В коде каждый генерирующий этап (`StageExecutor`) создаёт версию, которую редактор принимает вручную в side-by-side, а проверяющие этапы выдают замечания для точечного приёма. Чтобы сцепить этапы в фоне, пришлось бы авто-принимать результаты без участия человека — это размывает контроль качества, ради которого и строился продукт. Для single-user сценария ценность фоновой очереди не оправдывает усложнение и риск.

Impact:

Удалены реальные ветки кода: пункт `.queue` в `SidebarView` и заглушка раздела очереди в `RootView`. Сущностей/моделей данных очереди в коде не было (не реализованы) — миграции не требуются. Номера удалённых блоков в спеке (2.9, 2.11, 4.3, раздел 8) намеренно оставлены пропусками, чтобы сохранить перекрёстные ссылки. Исторические планы в `docs/superpowers/plans/` не переписываются. Инвариант «нет авто-публикации» сохранён и перенесён в раздел 9 спеки.

### 2026-06-20 — Отказ от опционального этапа «Стиль/Главред»

Status: active

Decision:

Опциональный этап «Стиль/Главред» убран из продукта. Он создавался под подключение внешнего сервиса «Главред», от которого решено отказаться. Из кода удалены поле `Topic.useStyle` и галочка «Использовать Стиль/Главред» в `BriefView`. Самого этапа в `PipelineStage` никогда не было — существовала только эта заготовка-флаг. Стиль и язык текста закрывает обязательный этап «Финальная вычитка» (агент ИИ-редактор). Видимых этапов теперь 9 обязательных + 1 опциональный (Продуктовые блоки); раньше было 9 + 2.

Why:

Этап имел смысл только как обёртка над внешним сервисом «Главред». Без него отдельная стилевая правка дублирует «Финальную вычитку» (та же роль ИИ-редактора, та же зона — язык и стиль). Один этап вместо двух проще для редактора и убирает мёртвую заготовку из модели данных и интерфейса.

Impact:

Спека обновлена до версии 8 (`docs/superpowers/specs/2026-05-19-content-system-redesign-design.md`), затронуты также frontend-design и checking-stages-design спеки. Удаление свойства `useStyle` из `@Model Topic` — лёгкая миграция SwiftData (поле просто исчезает у существующих тем). Исторические планы (`docs/superpowers/plans/*`) намеренно не переписаны — это записи уже выполненных под-проектов.



### 2026-06-23 — OAuth loopback: POSIX BSD-socket вместо NWListener

Status: active

Decision:

Для локального OAuth loopback-сервера (перехват redirect от Google) использовать обычный POSIX BSD-сокет (`socket/bind/listen/accept`) вместо `NWListener` из фреймворка Network. Привязка строго к `127.0.0.1`.

Why:

`NWListener` падает с EINVAL (POSIX 22) на macOS 26.x при любой конфигурации (`.any`, `.loopback`, явный порт, эфемерный порт). Корень проблемы в самом фреймворке, не в коде. POSIX-сокет работает штатно и даёт ту же loopback-only гарантию через `sin_addr = inet_addr("127.0.0.1")`.

Impact:

Файл `GoogleAuthService.swift` (класс `LoopbackListener`). Зависимость `import Network` убрана. Если на будущей macOS `NWListener` починят — можно вернуть, но тест будет не CLI-автоматизируемым (интерактивный браузер). Решение актуально для любых компонентов проекта, которым нужен локальный TCP-слушатель.

### 2026-06-20 — Галерея изображений отдельно от ленты версий; render-then-save split

Status: active

Decision:

Изображения (`GeneratedImage`) хранятся в отдельной галерее на `Topic` (связь `@Relationship(deleteRule: .cascade)`), не смешиваясь с `ArticleVersion`. `Topic.coverImageID` — UUID, указывающий на текущую обложку. Данные изображения хранятся с `@Attribute(.externalStorage)` (большие бинарные данные). Иллюстрация сохраняет `anchorQuote` (якорь-цитата) и `sourceImageID` (ссылка на исходник при доработке). Доработка только промтом: новая картинка создаётся как отдельная запись с `sourceImageID`, исходник не изменяется. Логика разделена на два независимых класса: `ImageGenerator` (рендер в `previewData`, лог `GenerationJob`) и `ImageSaver` (persist в галерею только по явному действию пользователя «Сохранить»).

Why:

Текст статьи и изображения — разные артефакты с разным жизненным циклом. Объединение в одну ленту загромоздило бы её и усложнило отображение. Разделение `ImageGenerator`/`ImageSaver` повторяет инвариант «человек принимает» и делает сетевой путь и путь сохранения тестируемыми независимо. `@externalStorage` обязателен для поля `data` — без него SwiftData хранит blob в основной БД, что замедляет запросы при большом числе изображений.

Impact:

Любой `ModelContainer`, регистрирующий `Topic`, должен также регистрировать `GeneratedImage`, `ImageStylePreset`, `ImagePromptTemplate` — иначе контейнер падает в runtime. Тестовые контейнеры в unit-тестах обновлены соответственно. Вставка изображений в Google Docs — будущий этап «Публикация» (не в объёме под-проекта 8); якорь-цитата сохранена как задел.

### 2026-05-24 — Этап «Структура» отдельно от текста; план как поле темы

Status: active

Decision:

ИИ-автор сначала строит план статьи, потом пишет текст. Добавлен отдельный первый этап пайплайна `PipelineStage.structure` (роль `author`): он генерирует план H1/H2/H3 с краткими пометками по разделам, без ключевых запросов. План хранится в `Topic.structureText` — редактируемом поле темы (по образцу `Topic.semantics`), а не как версия в ленте. Этап «Черновик» пишет текст по плану через новую переменную `{{структура}}`.

Why:

Раньше первый этап «Черновик» делал структуру и текст за один проход. Для длинных медицинских статей раздельный подход даёт лучший контроль (план — это SEO-каркас), дешёвую правку до генерации полного текста и человеческую контрольную точку, что согласуется с инвариантом «человек принимает». Хранение планом-полем выбрано вместо версии в ленте, потому что ручного редактирования версий в приложении пока нет, а план — рабочий вход, а не публикуемый текст; поле темы сразу редактируемо и повторяет готовый паттерн «Семантики».

Impact:

`StageTemplate` для `.draft` переписан на использование `{{структура}}` с мягким фолбэком (пустой план — пишем с нуля). Генерация структуры не создаёт `ArticleVersion`. Новый этап досевается автоматически (сидер добавляет недостающие этапы), миграция дефолтов не нужна. Изменение модели `Topic` — лёгкая миграция SwiftData (у старых тем `structureText` пустой). Семантика остаётся отдельным поздним этапом без ключей в структуре.

### 2026-05-23 — ИИ-роли и блоки контекста как редактируемые сущности

Status: active

Decision:

Роль ИИ-агента и общие методички хранятся отдельно от промтов этапов. В SwiftData добавлены `AIRole` (фиксированные ключи `author`, `seo`, `factChecker`, `editor`) и `ContextBlock` (фиксированные ключи `editorialPolicy`, `sources`, `seoGuidelines`). Этапы пайплайна имеют фиксированный `PipelineStage.roleKey`, а полный system prompt собирается как: мандат роли → тексты подключённых блоков в каноническом порядке → короткое дополнение этапа.

Why:

Раньше роль, редполитика, источники и SEO-правила дублировались в системных промтах этапов. Это усложняло правку и повышало риск расхождений. Отдельные роли и блоки дают правку в одном месте, но не расширяют скоуп до произвольного создания ролей или переназначения этапов.

Impact:

`StageTemplate.systemPrompt` теперь должен содержать только этап-специфичное дополнение, часто пустое. Пользовательский промт этапа, параметры модели и история запусков остаются отдельными. Новые запуски сохраняют имя роли как снимок в `GenerationJob.agentName` и `ArticleVersion.agentName`; старую историю не мигрируем.

### 2026-05-21 — Стек приложения: нативный SwiftUI

Status: active

Decision:

Приложение реализуется как нативное macOS-приложение на SwiftUI. Веб-обёртка (Tauri/Electron) отклонена.

Why:

Нужен только macOS — кроссплатформенность веб-обёртки не требуется. Один цельный стек (Swift + SwiftUI) проще вести не-разработчику с ИИ-сопровождением и надёжнее, чем связка «веб-фронт + упаковщик». Нативный вид и ощущение, хорошая поддержка ИИ. Целевые экраны (side-by-side, дерево Базы знаний, списки, формы) хорошо ложатся на SwiftUI.

Impact:

Skill frontend-design (веб-ориентированный) напрямую не применяется — UI проектируется через brainstorming + макеты и реализуется в SwiftUI. Потребуется Xcode и разовая настройка подписи приложения. Следующий шаг — фронтенд-дизайн ключевых экранов.

### 2026-05-21 — Продуктовая архитектура нового приложения (Вариант B)

Status: active

Decision:

Принят Вариант B (смелое переосмысление) для редизайна приложения. Ключевое: единая лента версий вместо слотов; side-by-side как основной экран; короткий видимый пайплайн (8 обязательных + 2 опциональных этапа), технические этапы скрыты в контекст ИИ-ролей; ИИ-роли (автор, SEO, фактчекер, редактор) с независимой проверкой; База знаний клиники (древовидный справочник); структурированный бриф (направление/врач из Базы знаний) вместо тегов; Google Docs только как точка публикации без обратной синхронизации; ручная правка сквозная.

Why:

Главная боль старого приложения — слишком много видимых этапов и непрозрачность артефактов. Вариант B убирает это, сохраняя полезную суть. Single-user снимает переходный барьер.

Impact:

Полная спека — `docs/superpowers/specs/2026-05-19-content-system-redesign-design.md`. Перед реализацией нужно выбрать стек (нативный macOS / веб). Отложено: автоимпорт из Docs, аналитика конкурентов.

### 2026-05-17 — Plan files are source of truth for plan-driven progress

Status: active

Decision:

For Superpowers or other plan-driven workflows, `docs/superpowers/plans/<plan>.md` is the source of truth for execution progress. Agents must update plan checkboxes after each completed plan task, add short `Note:` entries for local judgment calls, and use `Plan Task <N>: <short action>` commit messages for plan-driven commits.

Why:

Internal TodoWrite state is session-local and disappears when switching agents or chats. Keeping progress in the plan file and using a simple commit convention makes handoff verifiable without extra tools.

Impact:

Ordinary non-plan-driven tasks are unchanged. `ai/decisions.md` remains for durable decisions, `ai/changelog.md` remains for notable summaries, and plan notes are used only for local decisions inside a specific plan.
