# Режим «Быстрая проверка» (FT-20260623-004) — Design

Date: 2026-06-26
Status: approved-pending-user-review
Source future task: FT-20260623-004 (раздел 5 спеки, «Режим Быстрая проверка»)

## Цель

Отдельный лист для разовой проверки произвольного текста **без создания темы**.
Пользователь вставляет текст, выбирает одну из трёх проверок, получает список
замечаний с возможностью принять/отклонить и собранный исправленный текст.
Результат можно скопировать или сохранить как новую тему. Автоматически в базу
ничего не пишется.

## Объём (согласовано с пользователем)

- **UI:** лист (sheet) поверх главного экрана, точка входа — на экране списка тем.
- **Проверки:** ровно одна за запуск, выбор из трёх этапов вида `.checking`:
  Проверка SEO (`seoCheck`), Фактчекинг (`factCheck`), Финальная вычитка (`finalReview`).
- **Результат:** замечания (`Remark`) + принять/отклонить + исправленный текст
  через `RemarkApplier` — как в обычном пайплайне на этих этапах.
- **Кнопки:** «Скопировать результат» (исправленный текст в буфер) и
  «Сохранить как тему».
- **Сохранять как тему:** в новую тему кладётся **исправленный** текст (с применёнными
  принятыми замечаниями; если ничего не принято — равен исходному). **Имя темы вводит
  пользователь** в поле с предзаполнением по первой непустой строке текста (редактируемое).
- **Без изменений схемы SwiftData.**

### Вне объёма (YAGNI)

- Отдельное окно (window) — отложено; при необходимости отдельной задачей.
- Side-by-side diff для проверок — неприменим (этапы `.checking` не переписывают
  текст целиком, а выдают точечные замечания).
- Множественный выбор проверок за один запуск.

## Подход

**Вариант A (выбран):** добавить метод `executeQuickCheck` в существующий
`StageExecutor`, который выполняет проверку без персистентности (без `GenerationJob`,
без реальной `Topic`). Промт собирается существующим `PromptBuilder` через временный,
**не вставленный в контекст** `Topic`, у которого заполнен только вставленный текст
(`currentText`); остальные переменные пустые. Ответ парсится существующим
`RemarksParser`.

Отклонённые варианты:
- B (переиспользовать `execute()` через отдельную in-memory базу) — лишние движущиеся
  части, смешение контекстов при «Сохранить как тему».
- C (обобщить `execute()` под «с темой/без темы») — правит работающий код пайплайна,
  риск регрессии, избыточно.

## Компоненты

### `StageExecutor.executeQuickCheck(stage:pastedText:template:in:)` (новый метод)

- Что делает: гоняет один checking-этап на произвольном тексте, заполняет
  `self.remarks`, `self.isRunning`, `self.lastErrorMessage`/`lastWarningMessage`.
  **Не** создаёт `GenerationJob`, **не** создаёт `ArticleVersion`, **не** трогает тему.
- Как использует: тот же `streamProvider` / `keyProvider` / `fetchRole` / `buildRoleContext`.
  Промт — `PromptBuilder().build(template:topic:currentText:selectedBlocks:[]:roleContext:)`
  с транзитным `Topic(title: "", articleType: .info)`, `currentText = pastedText`.
- Зависит от: `PipelineStage` (kind == .checking), `StageTemplate` выбранного этапа,
  `ModelContext` (только чтение ролей/контекст-блоков).
- Предусловие: вызывается только для этапов `kind == .checking`.

### `Views/QuickCheckSheet.swift` (новый)

- Поля состояния: `inputText`, `selectedStage` (из трёх checking-этапов),
  `acceptedRemarkIDs`/`rejectedRemarkIDs`, `proposedTitle` (для сохранения).
- Тело: TextEditor ввода → Picker проверки → кнопка «Проверить» (зовёт
  `executeQuickCheck`) → `RemarksPanelView` (переиспользование) → строка кнопок.
- Исправленный текст: вычисляемое `RemarkApplier.apply(base: inputText, accepted: …)`.
- «Скопировать результат»: исправленный текст в `NSPasteboard`.
- «Сохранить как тему»: поле имени (предзаполнение первой строкой, редактируемое) →
  создать `Topic(title:, articleType: .info)`, вставить; создать
  `ArticleVersion(stage: selectedStage, source: .checkApplied, text: исправленный)`,
  привязать к теме, выставить `topic.currentVersionID`.
- Шаблон этапа: `@Query`/fetch `StageTemplate` для `selectedStage` (как в воркспейсе).

### Точка входа

Кнопка/пункт меню «Быстрая проверка» на экране списка тем (`RootView`/`ContentView`),
открывающая `QuickCheckSheet` через `.sheet`.

## Поток данных

```
вставленный текст + выбранная проверка
        │
        ▼
StageExecutor.executeQuickCheck ── streamProvider ──► ответ ИИ
        │                                              │
        │                                       RemarksParser.parse
        ▼                                              ▼
   self.remarks ────► RemarksPanelView (принять/отклонить)
                              │
                       RemarkApplier.apply(base: вставленный, accepted)
                              ▼
                       исправленный текст
                         ├── «Скопировать» → NSPasteboard
                         └── «Сохранить как тему» → новый Topic + ArticleVersion
```

## Обработка ошибок

- Нет API-ключа → то же сообщение, что и в `execute()` («Укажите API-ключ в Настройках»).
- Пустой ввод → кнопка «Проверить» неактивна.
- Обрыв по лимиту токенов → `lastWarningMessage` (как в `execute()`).
- ИИ вернул некорректный JSON → `RemarksParser` отдаёт пустой список; показать «Замечаний нет».

## Тестирование

- `QuickCheckTests` (новый): `executeQuickCheck` с мок-`StreamProvider` (по образцу
  `FragmentEditTests`) — из валидного JSON парсятся замечания; ничего не пишется в контекст
  (нет новых `GenerationJob`/`ArticleVersion`).
- Логика «исправленный текст» уже покрыта тестами `RemarkApplier` — отдельно проверить
  предзаполнение имени по первой строке (чистая функция-хелпер).
- Сборка: `xcodebuild build-for-testing` зелёный; тесты — Cmd+U.

## Риски

- Изменений схемы SwiftData нет → миграции нет.
- Главный нюанс — транзитный несохранённый `Topic` для сборки промта: убедиться, что
  `PromptBuilder` корректно отрабатывает пустые связи (`attachedNodes`, `direction`,
  `doctor`, `semantics`) на свежем `Topic`. Покрывается тестом сборки промта.
- `executeQuickCheck` не должен случайно вставлять `GenerationJob` — отдельный метод,
  не трогающий персистентность.
