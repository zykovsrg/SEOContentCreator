# Дизайн: загрузка иллюстраций на Google Диск + раздел «Техническая информация»

Дата: 2026-07-20
Статус: утверждён пользователем (чат 2026-07-20)

## Цель

Две связанные доработки публикационного конвейера:

1. Иллюстрации статьи загружаются на Google Диск в подпапку статьи при публикации.
2. После принятия версии этапа «Финальная вычитка» в конец текста статьи
   дописывается раздел `## Техническая информация` по фиксированному шаблону.

Поддерживающее изменение: у темы появляются дополнительные направления
(мультивыбор из Базы знаний с созданием нового направления на месте).

## Решения, принятые с пользователем

- Подпапка на каждую статью (не общая свалка).
- Загрузка происходит при публикации; пользователь галочками выбирает обложку
  и нужные картинки.
- Раздел «Техническая информация» живёт в тексте статьи (не только в Google Doc).
- «Раздел» выводится из существующего `ArticleType`:
  `disease → /deseases/`, `service → /services/`, `info → /article/`
  (написание путей — как на сайте пользователя). Новое поле темы не нужно.
- «Направления»: основное направление темы + новые дополнительные направления
  (мультивыбор из Базы знаний, с возможностью создать направление при выборе).
- OAuth-права Google НЕ расширяются (вариант Б): остаёмся на `drive.file`.
  Папки для картинок создаются внутри папки приложения «SEO-статьи клиники»,
  которую приложение создало само и потому видит. Указанная пользователем
  внешняя папка (`1XXEa9SH51qKIbUUHrZNDfdsrXDZ3LPlB`) НЕ используется —
  пользователь осознанно выбрал папку приложения, чтобы не расширять права
  и не проходить повторный вход в Google.

## 1. Загрузка иллюстраций на Google Диск

### Структура папок

```
SEO-статьи клиники/            ← существующая папка приложения (создаётся findOrCreateFolder)
  Иллюстрации/                 ← создаётся при первой загрузке
    №[externalID] [Тема]/      ← подпапка статьи; имя — как заголовок публикации (PublishTitleBuilder)
      обложка.png
      иллюстрация-1.png
      иллюстрация-2.png
```

### Поток публикации

1. В окне публикации появляется шаг «Картинки»: сетка неархивных
   `GeneratedImage` темы с чекбоксами; обложка (`topic.coverImageID`)
   отмечена заранее.
2. Публикация документа идёт как раньше (`ArticlePublisher`).
3. После успешной публикации выбранные картинки загружаются в подпапку статьи.
4. Ссылка на подпапку (`webViewLink`) сохраняется у темы
   (`Topic.illustrationsFolderURL: String?`) и подставляется в строку
   «Иллюстрации:» публикуемого документа (см. раздел 2).

### Без дублей

`GeneratedImage.driveFileID: String?` — ID уже загруженного файла.
При повторной публикации картинки с непустым `driveFileID` пропускаются.
Если пользователь удалил файл на Диске вручную, пропуск оставит картинку
незагруженной — это принятое ограничение (решается очисткой поля в будущем,
вне scope).

### Новый код

- `GoogleDocsClient`:
  - `findOrCreateFolder(name:parentID:)` — как существующий
    `findOrCreateFolder(name:)`, но с поиском/созданием внутри родителя
    (query `'parentID' in parents`).
  - `uploadFile(name:data:mimeType:parentID:) -> (fileID: String, webViewLink: String?)` —
    multipart upload `POST /upload/drive/v3/files?uploadType=multipart`.
- `ImageDriveUploader` (новый модуль в `Logic/`): оркестрирует
  «найти/создать цепочку папок → загрузить выбранные → проставить driveFileID →
  вернуть ссылку на подпапку». Протокол-зависимость от клиента для тестов
  (по образцу `ArticlePublisher`).

### Ошибки

- Ошибка загрузки картинок НЕ откатывает публикацию документа: документ уже
  опубликован, пользователю показывается понятное сообщение «Документ
  опубликован, но картинки загрузить не удалось: …», повторная публикация
  дозагружает недостающее.

## 2. Раздел «Техническая информация»

### Шаблон (Markdown, в конце текста статьи)

```
## Техническая информация

Тайтл: <ArticleVersion.seoTitle | «[вписать вручную]»>
Дескрипшн: <ArticleVersion.seoDescription | «[вписать вручную]»>
Эксперт: <topic.doctor?.title | «[вписать вручную]»>
Врачи отделения: [вписать вручную]
Направления: <основное + дополнительные направления через запятую | «[вписать вручную]»>
Раздел: </deseases/ | /services/ | /article/>
Иллюстрации: [появится при публикации]
```

### Когда дописывается

В момент принятия версии этапа «Финальная вычитка» (accept-путь в
`VersionActions`): к принимаемому тексту добавляется раздел, если заголовка
`## Техническая информация` в тексте ещё нет (идемпотентность — повторное
принятие не создаёт второй раздел). Новая версия при этом не создаётся —
раздел входит в принимаемый текст.

### Подстановка ссылки на иллюстрации

- В тексте внутри приложения строка остаётся
  `Иллюстрации: [появится при публикации]`.
- При публикации, если у темы есть `illustrationsFolderURL`, строка в
  публикуемом документе заменяется на реальную ссылку (замена значения строки
  на этапе подготовки текста к публикации, до `DocsRequestBuilder`).
- Ссылка также видна в приложении рядом с данными публикации.

### Новый код

- `TechInfoSectionBuilder` (новый файл в `Logic/`, чистая логика):
  - `build(topic:version:) -> String` — собирает раздел;
  - `append(to:topic:version:) -> String` — дописывает с проверкой на дубль;
  - `sectionPath(for: ArticleType) -> String` — маппинг типа на раздел сайта;
  - `substituteIllustrationsLink(in:url:) -> String` — замена заготовки на ссылку.

## 3. Дополнительные направления темы

- `Topic`: новое отношение
  `@Relationship var additionalDirections: [KnowledgeNode]` (nullify,
  дефолт `[]`). Существующее одиночное `direction` не меняется — его
  используют промты (`{{направление}}`), валидация брифа и подсчёт
  использования узлов.
- `BriefView`: мультивыбор дополнительных направлений (узлы `nodeType == .direction`,
  кроме уже выбранного основного) + кнопка «Создать направление» прямо в
  выборе (создаёт узел Базы знаний и сразу отмечает его).
- `KnowledgeNodeUsage`: учитывать `additionalDirections` при подсчёте
  использования узла.
- Миграция: лёгкая автоматическая (новое отношение с пустым дефолтом),
  существующие данные не трогаются.

## 4. Тесты

Юнит-тесты (чистая логика, без сети):

- `TechInfoSectionBuilder`: полный набор полей / пустые источники /
  идемпотентность append / маппинг раздела / подстановка ссылки.
- `ImageDriveUploader` с мок-клиентом: цепочка папок, пропуск уже
  загруженных, проставление `driveFileID`, возврат ссылки.
- `GoogleDocsClient.uploadFile` — сборка multipart-запроса (по образцу
  существующих тестов клиента, если они есть; иначе — через протокол-мок
  на уровне uploader).

Ручная проверка после реализации (реальный Google-аккаунт):

- публикация с выбором картинок → подпапка создана, файлы на месте;
- повторная публикация → дубли не создаются;
- строка «Иллюстрации:» в Google Doc содержит рабочую ссылку;
- раздел «Техническая информация» корректно выглядит в Google Doc.

## Вне scope

- Использование внешней пользовательской папки Drive (потребует полного
  scope `drive` и повторного входа) — осознанно отклонено.
- Автозагрузка картинок сразу после генерации.
- Очистка/повторная синхронизация `driveFileID` при ручном удалении файлов
  на Диске.
- Несколько основных направлений в промтах генерации.
