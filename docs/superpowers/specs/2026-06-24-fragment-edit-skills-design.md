# Дизайн: правка фрагмента — скиллы и регенерация (FT-20260623-003 пп.1–2)

Дата: 2026-06-24
Статус: согласован пользователем 2026-06-24

## Контекст

Из раздела 7 спеки редизайна не реализованы сквозные инструменты ручной правки на
выделенном фрагменте текста в рабочем пространстве темы. Пункт 3 того же FT
(мягкие алгоритмические подсказки) уже завершён и в `main`. Здесь — пункты 1 и 2:

1. **Пресеты скиллов** — библиотека мини-промтов для фрагмента («переписать в
   инфостиле», «упростить», «уточнить», «убрать канцелярит»), редактируется в
   «Шаблонах».
2. **Регенерация фрагмента** — выделить кусок, ввести комментарий «что не
   нравится», ИИ-автор переписывает только этот фрагмент.

Оба инструмента опираются на одну цепочку: получить фрагмент → прогнать ИИ только
на нём → side-by-side → принять → новая версия, где изменён только этот кусок.

## Согласованные решения

- **Выделение фрагмента — Вариант A (ручная вставка + автоподстановка из буфера).**
  Текст текущей версии в рабочем пространстве показывается только для чтения
  (`HighlightedText` с `.textSelection(.enabled)`), SwiftUI не отдаёт программно
  выделенный диапазон. Поэтому пользователь выделяет текст в статье, копирует
  (Cmd+C), открывает окно — поле фрагмента подставляется из буфера обмена и
  редактируемо. Выделение мышью на месте (AppKit NSTextView) — возможный UX-апгрейд
  позже, ядро (ИИ на фрагменте → версия) при этом не меняется.
- **Пресеты скиллов — средний вариант:** название + промт + выбор роли
  (редактор / автор / SEO), без персональных параметров модели. Используется
  существующая система ролей (`AIRole` + `RoleContextAssembler`).
- **UX — одно окно** «Правка фрагмента» с переключателем режима: «Скилл» (список
  пресетов) или «Свой комментарий» (поле «что не нравится»). Общее поле фрагмента.

## Поток

1. Пользователь выделяет текст в статье → Cmd+C → кнопка «Правка фрагмента» в тулбаре.
2. Окно открывается, поле фрагмента подставлено из буфера обмена (редактируемо).
3. Режим «Скилл»: выбрать пресет из списка. Режим «Свой комментарий»: ввести текст.
4. «Применить» → `FragmentPromptBuilder` собирает промт → `FragmentEditor` стримит
   переписанный фрагмент (виден потоково).
5. На завершении → `FragmentSplicer` вставляет фрагмент в полный текст текущей версии.
6. Side-by-side (новый полный текст против текущего) → «Принять» создаёт новую
   версию и делает её текущей; «Отклонить» — ничего не сохраняется.

## Замена фрагмента

`FragmentSplicer` — чистая функция. Поиск по точному совпадению подстроки в полном
тексте текущей версии:

- ровно 1 совпадение → `.replaced(newFullText)` (заменяется только это вхождение);
- 0 совпадений → `.notFound` → сообщение «фрагмент не найден в тексте — проверьте,
  что скопировали точно»;
- N > 1 совпадений → `.ambiguous(N)` → сообщение «фрагмент встречается N раз —
  расширьте выделение, чтобы он был уникальным».

При `.notFound` / `.ambiguous` версия не создаётся, ошибка показывается в окне.

- `Views/TemplatesView.swift` — раздел «Скиллы»: CRUD пресетов (название, промт,

### Роли

- Скиллы — каждый несёт свой `roleKey` (по умолчанию `editor`).
- Регенерация — всегда роль `author` (ИИ-автор), по спеке.

## Обработка ошибок

- Нет API-ключа → «Укажите API-ключ в Настройках» (как в `StageExecutor`).
- Сетевая ошибка → `error.localizedDescription` (как в `StageExecutor`).
- Обрыв по лимиту токенов → предупреждение (как в `StageExecutor`).
- Замена: `.notFound` / `.ambiguous(N)` → сообщения выше, версия не создаётся.

## Тестирование

- `FragmentSplicer`: `.replaced` при единственном совпадении (меняется только оно),
  `.notFound`, `.ambiguous(N)`, чувствительность к пробелам.
- `FragmentPromptBuilder`: system берётся из роли; user содержит инструкцию,
  фрагмент и указание «только фрагмент».
- `SkillPresetDefaults` / `SkillPresetSeeder`: засевает 4 пресета, идемпотентно,
  сброс к стандартному.
- `FragmentEditor` (мок `StreamProvider`): успех → создаётся версия со склеенным
  текстом и правильным `VersionSource`; путь ошибки; `.ambiguous` → версия не
  создаётся, ошибка видна.
- `VersionSource`: корректные title новых кейсов.

## Риски и совместимость

- Добавление `@Model SkillPreset` в схему — лёгкая миграция SwiftData (новая
  сущность, без явного `MigrationPlan`); существующие данные не теряются, как было
  с `EditorDictionary` в п.3. Любой тестовый `ModelContainer` должен регистрировать
  `SkillPreset`.
- Замена фрагмента по строке — снимается проверкой уникальности (`FragmentSplicer`).
- Выделение мышью на месте (AppKit) намеренно не входит в объём; остаётся
  возможным UX-апгрейдом, не меняющим ядро.

## Вне объёма

- Выделение произвольного диапазона мышью прямо в статье (Вариант B).
- Персональные параметры модели у скилла (модель / температура / reasoning_effort).
- Пакетное применение скилла к нескольким фрагментам.
