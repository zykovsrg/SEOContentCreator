Это дизайн под-проекта 4 — проверяющие этапы поверх «Ядра генерации». Описывает логику, сущности и UI, без кода. Реализация — отдельным планом (writing-plans).

## 1. Концепция

Три проверяющих этапа поверх готового пайплайна генерации: **Проверка SEO**, **Фактчекинг**, **Финальная вычитка**. В отличие от авторских этапов (которые пишут текст), проверяющие **не переписывают текст молча** — они возвращают **список замечаний** (карточки «что не так и почему»), а редактор принимает правки **точечно, по каждому замечанию**. У каждого проверяющего — свой ИИ-агент со своей «методичкой» и придирчивым мандатом (см. продуктовую спеку §2.14: ценность проверок — в разном мандате, входе и знаниях).

## 2. Принятые решения (из brainstorming 2026-05-22)

1. **Приём правок — Вариант B (точечно по замечаниям).** У каждого замечания своя кнопка ✓/✗; принятие применяет конкретную замену фрагмента. (Альтернатива — приём целиком/по абзацам — отклонена как менее точная для медтекста.)
2. **Контекст агентов — Вариант A (стартовые «методички» в промтах).** SEO-правила, редполитика, принципы источников зашиты как стартовые тексты в промты этапов. Отдельный редактор — позже, в разделе «Шаблоны». (Альтернатива — отдельное редактируемое хранилище сейчас — отклонена: пересекается с будущими «Шаблонами».)
3. **Версии — Подход 1 (одна версия по «Готово»).** Принятие карточек применяется к рабочей копии вживую; одна новая версия создаётся при нажатии «Готово». (Альтернатива — версия на каждое принятие — отклонена: засоряет ленту.)

## 3. Этапы и агенты

Добавляются 3 этапа к существующим (Черновик, Продуктовые блоки, Семантика-в-текст):

| Этап (raw) | Заголовок | Агент | Что проверяет |
|---|---|---|---|
| `seoCheck` | Проверка SEO | ИИ-SEO | заголовки H1/H2/H3, ключи и плотность, Title/Description, объём, рекламность |
| `factCheck` | Фактчекинг | ИИ-фактчекер | имена врачей, процедуры, цены, доказательные утверждения — сверка с Базой знаний |
| `finalReview` | Финальная вычитка | ИИ-редактор | орфография, грамматика, пунктуация, стиль, канцелярит, «вода» |

Изменения модели `PipelineStage`:
- Добавить кейсы `seoCheck`, `factCheck`, `finalReview`.
- Добавить признак типа этапа: `kind: StageKind` (`.author` / `.checking`).
- `agentName` зависит от этапа: авторские → «ИИ-автор»; `seoCheck` → «ИИ-SEO»; `factCheck` → «ИИ-фактчекер»; `finalReview` → «ИИ-редактор».
- Порядок этапов в полосе (по продуктовой спеке §4.0): Создание (Черновик, Продуктовые блоки) → Оптимизация (Семантика-в-текст) → Контроль (Проверка SEO, Фактчекинг, Финальная вычитка). Группировка в UI — желательна, но не обязательна для v1 (допустим простой ряд из 6 чипов).

## 4. Замечание (Remark) и формат вывода

Проверяющий ИИ возвращает **только список замечаний** (никакого полного переписывания), строго в JSON. Структура замечания:
- `category` — категория (строка, напр. «Канцелярит», «Плотность ключей», «Факт», «Title/Description»).
- `quote` — точный фрагмент исходного текста, к которому относится замечание (для поиска и подсветки).
- `suggestion` — предлагаемая замена фрагмента (может быть пустой для замечаний-предупреждений без авто-правки).
- `explanation` — что не так и почему.

Формат ответа модели (после потокового текста): блок
```json
{"remarks":[{"category":"…","quote":"…","suggestion":"…","explanation":"…"}]}
```

`Remark` — это **транзитная структура** (Swift `struct`, `Codable`), не сущность SwiftData: проверка stateless, при повторном запуске замечания пересоздаются. Парсер `RemarksParser` извлекает массив `remarks` из ответа (по аналогии со `StageOutputParser` из под-проекта 3; устойчив к отсутствию/битому JSON — тогда пустой список).

В UI к каждому замечанию добавляется состояние приёма (`accepted` / `rejected` / не решено) — это состояние представления, не хранится.

## 5. Выполнение проверки

Расширение `StageExecutor` (или родственный путь исполнения) для проверяющих этапов:
- Стримит ответ модели (как сейчас), показывает прогресс.
- По завершении: для **проверяющего** этапа парсит замечания в **транзитное** состояние `remarks: [Remark]` (новое наблюдаемое свойство) и **не создаёт версию**. Для авторских этапов поведение прежнее (создаёт версию-предложение).
- Запись в лог (`GenerationJob`) создаётся как обычно (этап, агент, модель, статус, ошибка). Можно сохранить число замечаний в `GenerationJob` (опционально).
- **Фактчекеру** в промт дополнительно подставляются прикреплённые узлы Базы знаний: расширить `PromptBuilder` переменной `{{база_знаний}}` (содержимое прикреплённых узлов темы: преимущества, данные врача, факты). Остальные переменные переиспользуются.

Стартовые шаблоны 3 проверяющих этапов добавляются в `StageTemplateSeeder` (с зашитыми методичками в `systemPrompt` и инструкцией вернуть JSON-замечания в `userPromptTemplate`).

## 6. Панель замечаний и приём (Вариант B + Подход 1)

UI рабочего пространства для проверяющего этапа:
- **Основная область** — текст текущей версии (рабочая копия), с возможностью подсветки фрагментов.
- **Панель справа** — карточки замечаний: категория, пояснение, «было → станет», кнопки ✓ принять / ✗ отклонить. Клик по карточке → подсветка и прокрутка к `quote` в тексте.
- **Принятие** замечания применяет `quote → suggestion` к рабочей копии **вживую** (видно изменение). Отклонение убирает карточку из активных.
- **Применение правок** — чистая тестируемая функция `RemarkApplier.apply(base:accepted:) -> String`: применяет принятые замены к базовому тексту с учётом порядка и повторяющихся фрагментов (заменяет вхождения по позиции, слева направо, не пересоздавая уже изменённое). Если `quote` не найден (текст изменился) — замечание помечается «не удалось применить», пропускается.
- **«Готово»** создаёт **одну** новую `ArticleVersion` с рабочей копией (источник — новый кейс `VersionSource.checkApplied`), делает её текущей, очищает замечания. Если принятых правок нет — новая версия **не создаётся**, этап считается пройденным (фиксируется через лог/последний успешный этап).
- **«Отклонить всё» / закрыть** — сбрасывает замечания без изменений.

Переиспользуется из под-проекта 3: лента версий, лог темы, стриминг, рабочее пространство темы, экран настроек.

## 7. Сущности и изменения (сводка)

- `PipelineStage` — +3 кейса, `kind`, поэтапный `agentName`.
- `VersionSource` — +кейс `checkApplied` («Правки проверки»).
- `Remark` — новая транзитная `struct` (Codable).
- `RemarksParser` — новый (парсинг JSON-замечаний).
- `RemarkApplier` — новый (применение принятых замен; чистая логика, тестируемая).
- `StageExecutor` — ветка для проверяющих этапов (замечания вместо версии); `remarks` свойство.
- `PromptBuilder` — +переменная `{{база_знаний}}`.
- `StageTemplateSeeder` — +3 шаблона проверяющих этапов с методичками.
- UI — панель замечаний (карточки + подсветка), кнопка «Готово»; интеграция в `TopicWorkspaceView` и `StageBarView`.
- `GenerationJob` — опционально число замечаний.

## 8. Тестирование

- **Unit (TDD):** `PipelineStage` (агенты/типы новых этапов), `RemarksParser` (валидный/битый/пустой JSON), `RemarkApplier` (одно замечание, несколько, дубли фрагментов, ненайденный фрагмент, пустой список), `PromptBuilder` (`{{база_знаний}}`), `StageTemplateSeeder` (6 шаблонов), `StageExecutor` (проверяющий этап кладёт замечания и не создаёт версию; ошибка → лог-ошибка).
- **Ручной smoke:** реальный прогон каждого проверяющего этапа на сгенерированной статье; приём/отклонение карточек; подсветка по клику; «Готово» создаёт одну версию; фактчекинг учитывает Базу знаний.

## 9. Границы (scope)

В скоупе: 3 проверяющих этапа, агенты с зашитыми методичками, формат и парсинг замечаний, панель карточек с точечным приёмом, живой просмотр, одна версия по «Готово», подстановка Базы знаний фактчекеру, записи в логе, unit-тесты + smoke.

Вне скоупа (на будущее): редактор методичек/промтов (раздел «Шаблоны»), этап «Стиль», публикация в Google Docs, полная сущность «Семантика», очередь, мягкие алгоритмические подсказки при ручной правке, пословный diff.
