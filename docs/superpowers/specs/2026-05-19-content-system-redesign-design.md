## 1. Краткая концепция нового приложения

Новое приложение — это персональная **писательская студия** для одного редактора, которая ведёт каждую тему по рекомендованному (но не жёсткому) пайплайну от идеи до публикации, а всю «техническую кухню» (редполитика, работа с источниками, SEO-рекомендации, базовая структура) прячет в контекст специализированных ИИ-ролей. Главный рабочий экран темы — это **две колонки side-by-side**: слева текущая версия текста, справа новая с подсвеченными правками, которые можно принять целиком или частично; ручная правка доступна на любом этапе. Над текстом работают несколько **ИИ-агентов с закреплёнными ролями** (автор, SEO-специалист, фактчекер, редактор): у проверяющих ролей свои задачи, вход и знания, отдельные от автора — это даёт более строгую проверку. Любой этап можно запустить в любой момент. Статус темы предельно простой: «докуда дошли + когда» плюс метки «идёт / ошибка». Очередь живёт компактно в сайдбаре, работает последовательно, корректно останавливается и умеет повторять только упавшее. Для разовых задач есть режим «Быстрая проверка» без создания темы. Google Docs — только финальная точка публикации, без обратной синхронизации.

## 2. Основные сущности

Здесь — только **сущности**: объекты данных, которые хранятся и имеют свойства. Действия над текстом (этапы) описаны в разделе 5.

### 2.1 Тема (Topic)
Центральная сущность. Сквозной ID, живёт от создания до публикации (или архивации).

Атрибуты:
- ID (стабильный, не меняется при правках)
- Название
- Тип статьи (см. 2.4)
- Целевой объём
- Ссылка на внешний документ (заполняется при публикации; см. 2.12)
- Указатель на текущую (активную) версию текста
- Отметка «последний успешно завершённый этап» + дата (основа статуса, см. 3)
- Дата создания, дата последнего изменения
- Флаги: «использовать Стиль/Главред», список выбранных продуктовых блоков
- Направление — узел Базы знаний (обязательно перед Черновиком; см. 2.13)
- Врач — узел Базы знаний (опционально; см. 2.13)
- Прикреплённые узлы Базы знаний клиники — дополнительные справочные данные для темы (см. 2.13)

Тема — это контейнер. Сам текст хранится в **версиях** (см. 2.3), привязанных к теме.

### 2.2 Семантика (Semantics)
Отдельная сущность, привязанная 1:1 к теме.

Атрибуты:
- Список запросов
- У каждого запроса: текст, частотность, статус («включён», «исключён», «обязательный»)
- Опциональная группировка (например «головные», «уточняющие», «продуктовые»)
- Опциональный импорт из Топвизора — подгрузка кластеров и частотностей (внешняя интеграция, выключаемая)

Семантика живёт отдельно от текста: её можно собирать и чистить до и после генерации. Используется на этапе «Семантика-в-текст» и как часть контекста промтов. Факт «собрана / не собрана» выносится в статус темы отдельным индикатором.

### 2.3 Версия текста (Version) — единая лента
**Упрощение:** нет отдельных «слотов-артефактов». У темы есть **одна общая лента версий текста**. Это как история сохранений: каждое явное действие добавляет новый снимок текста в ленту, ничего не затирая.

У каждой версии есть:
- ID, временная метка
- **Метка этапа**, на котором она создана (черновик / продуктовые блоки / семантика-в-текст / стиль / проверка SEO / фактчекинг / финальная вычитка / ручная правка). Структура — не версия, а редактируемое поле темы.
- **Источник**: «сгенерировано системой», «ручная правка», «принятие правок (полное / частичное)», «откат», «импорт из Docs» (+ какой агент, если генерация — см. 2.14)
- Сам текст (+ метаданные: H1, Title, Description, когда они появляются)
- Какой шаблон промта, агент и модель использовались (если генерация)
- Заметка автора (опционально)
- Признак «текущая версия» (голова ленты — то, что считается истиной сейчас)

Правила:
- Версии **никогда не удаляются** автоматически. Можно «архивировать» (скрыть из обычного списка), но не удалить.
- **Откат** — это назначение старой версии текущей; при этом в ленту добавляется новая запись «откат к версии X», сама старая версия остаётся в истории.
- **Два вида отображения ленты** (переключатель): «по времени» — хронология всех версий; «по этапам» — версии сгруппированы по меткам этапа (выглядит как «слоты-ячейки»). Под капотом это одна лента, поэтому нет дублирования и всегда ясно, какая версия текущая.
- «Финал» — это не отдельный слот, а текущая версия с пройденными этапами «Проверка SEO», «Фактчекинг» и «Финальная вычитка».

### 2.4 Тип статьи (Article Type)
Классификатор, который определяет:
- системный контекст (медицинские правила, тон, ограничения)
- какие этапы обязательны/опциональны
- какие шаблоны промтов применяются
- какие продуктовые блоки уместны
- какие SEO-рекомендации подставляются

Стартовый набор: Заболевание, Услуга, Информационная статья. Редактируется в разделе «Шаблоны».

### 2.5 Шаблон этапа (Stage Template)
Шаблон промта для конкретной пары (этап × тип статьи). Выполняется агентом, ответственным за этот этап (см. 2.14): контекст агента подставляется автоматически.

Структура шаблона (простыми словами — «заготовка инструкции для ИИ с пустыми местами под данные темы»):
- **Системный контекст** — общие правила «кто ты и как себя ведёшь»: медицинская аккуратность, доказательная осторожность, запрет на искажение фактов, правила читабельности. Версионируется.
- **Шаблон этапа** — что именно сделать на этом шаге.
- **Правила подстановки (переменные)** — в шаблон вставляются переменные `{{…}}`, подставляемые автоматически. Каждая переменная определяется **один раз в своём источнике** и переиспользуется во всех шаблонах. Реестр переменных и источники:
  - из Брифа: `{{тема}}`, `{{тип}}`, `{{объём}}`, `{{направление}}`, `{{врач}}`
  - из Семантики: `{{семантика}}`
  - из Базы знаний (по прикреплённым узлам): `{{преимущества}}`, `{{врач_данные}}`, `{{источники_направления}}`
  - из предыдущей версии (лента): `{{текущий_текст}}`
  - из проверок / внешних сервисов: `{{замечания}}`
- **Специальные правила по типу статьи** — включаются, если тип совпадает.
- **Параметры модели** — какая модель и её настройки.

**Что такое «температура» (для справки):** настройка «насколько ИИ креативит». Низкая = ровный, предсказуемый текст. Высокая = разнообразнее, но менее предсказуемо. Для медицины — низкая-средняя. Параметры задаются в приложении и передаются в запрос к ИИ через API (это «способ программно обращаться к ИИ-сервису»). Параметры модели — опция расширенного режима.

Любая правка шаблона создаёт его новую версию (для воспроизводимости старых тем).

### 2.6 Скрытый контекст: редполитика, источники, SEO-рекомендации
Это **не видимые этапы**, а переиспользуемые куски контекста, которые закрепляются за ИИ-ролями (см. 2.14) и подставляются в их промты. Живут в разделе «Шаблоны».

- **Редполитика** — медицинские и редакционные правила.
- **Принципы работы с источниками** — как опираться на доказательность; дополняется приоритетными ссылками-источниками из Базы знаний по направлению темы (см. 2.13).
- **SEO-рекомендации** — правила по заголовкам (H1/H2/H3), Title/Description, структуре, объёму, плотности ключей, ограничениям на рекламность. Эталон для этапа «Проверка на соответствие SEO».

Редактор их каждый день не трогает. При необходимости правит в «Шаблонах».

### 2.7 Продуктовый блок (Product Block)
Каждый продуктовый блок (преимущество клиники, CTA, блок «почему мы», блок врача и т.п.) — это **отдельный шаблон** в библиотеке блоков. Данные для блоков (имена врачей, преимущества, факты) берутся из прикреплённых узлов Базы знаний клиники (см. 2.13). Как блоки выбираются и встраиваются — этап «Продуктовые блоки» (раздел 5).

### 2.8 Пресет «скилл» для фрагмента (Fragment Preset)
Мини-промт для выделенного куска текста: «перепиши в инфостиле», «упрости», «уточни», «убери канцелярит» и т.п. Это **редактируемая библиотека пресетов**, доступная при сквозной ручной правке на любом этапе (см. раздел 7). Применение пресета показывается как локальный side-by-side (было/стало) и принимается точечно.

### 2.9 Сценарий запуска (Run Scenario)
Конфигурация запуска: **до какого этапа дойти и где остановиться** для ручного вмешательства. Пресеты: «до черновика», «до семантики-в-текст», «до проверок», «полный» (останавливается перед публикацией). Плюс **свободный выбор этапов** — по умолчанию доступен.

Жёсткое правило: автоматической публикации без явного клика «опубликовать» не бывает.

### 2.10 Запуск (Job/Run)
Один конкретный запуск этапа или сценария — «история действия». Атрибуты: когда начат/завершён, какой этап / агент / шаблон / модель, стоимость (токены — опц.), результат (успех/ошибка/отменён). Успешный запуск добавляет версию в ленту. Запуски видны в логе темы и в логе очереди.

### 2.11 Очередь (Queue)
Список тем на фоновую последовательную обработку. Состояние: «остановлена / работает / останавливается после текущей». Хранит активную позицию и лог (начато / завершено / упало с сообщением). Живёт компактно в левом сайдбаре + полный раздел. Поведение — раздел 8.

### 2.12 Внешний документ (External Document)
Ссылка на опубликованный документ Google Docs + дата публикации + история всех публикаций темы. Хранится в теме. **Без обратной синхронизации.** Возврат правок из Docs — только вручную (copy-paste как новая версия с меткой «импорт из Docs»). Механика публикации — раздел 9; этап «Публикация» — раздел 5.

### 2.13 База знаний клиники (Clinic Knowledge Base)
Структурированный **древовидный справочник** фактов о клинике. Редактор сам строит и меняет дерево любой вложенности и наполняет его содержимым.

Пример структуры:
- 1. Общие преимущества клиники
- 2. Направления
  - 2.1 Лучевая терапия
    - 2.1.1 Преимущества
      - 2.1.1.1 Лечение рака простаты за одно посещение клиники
  - 2.2 Врачи
    - 2.2.1 Усычкин Сергей Владимирович

Устройство:
- **Узел дерева:** заголовок + (опционально) содержимое (факт, описание, данные врача). Узлы вкладываются друг в друга.
- Редактируются и **структура** (добавить / удалить / переместить / переименовать узел), и **содержимое**.
- Живёт в отдельном разделе приложения (см. 4.6).

Связь с темой — **ручное прикрепление + умное предложение:**
- К теме редактор прикрепляет нужные узлы (например, «Лучевая терапия» + врач «Усычкин»). Прикреплённые узлы хранятся в теме (см. 2.1).
- Система **подсказывает** релевантные узлы по названию и типу темы, но финальное решение — за редактором.

Три роли в работе:
1. **Черновик** (агент Автор) — прикреплённые узлы подставляются в промт, чтобы ИИ не выдумывал имена и процедуры.
2. **Продуктовые блоки** (агент Автор; см. 2.7) — берут данные из прикреплённых узлов (блок врача, преимущества).
3. **Фактчекинг** (агент Фактчекер; раздел 5) — сверяет факты в тексте с прикреплёнными узлами.

**Ссылки-источники по направлению.** У узла-направления можно указать приоритетные внешние источники — ссылки на сайты профессиональных сообществ, методички. Агент опирается на них в первую очередь; это связано со скрытым контекстом «Принципы работы с источниками» (см. 2.6).

**Фильтрация.** По справочнику доступны срезы: например, «показать все преимущества отделений лучевой терапии и урологии». Фильтрация — по ветке/направлению и по типу узла.

### 2.14 ИИ-роль (агент)
**Специализированный ИИ-исполнитель** с закреплённой ролью и собственным контекстом («методичкой»), который автоматически подставляется при каждом его запуске. Технически модель не «помнит» между запусками — продуктово это значит «у роли всегда с собой её знания».

Атрибуты роли:
- Имя/роль
- **Что знает** — закреплённые куски скрытого контекста (см. 2.6) и/или База знаний (см. 2.13)
- **Этапы**, за которые отвечает

Стартовый состав:

| Агент | Что знает (контекст) | Этапы |
|---|---|---|
| **ИИ-автор** | редполитика, работа с источниками, продуктовые блоки, встраивание семантики, метатеги | Структура, Черновик, Продуктовые блоки, Семантика-в-текст |
| **ИИ-SEO** | SEO-требования | Проверка SEO |
| **ИИ-фактчекер** | принципы работы с источниками, База знаний | Фактчекинг |
| **ИИ-редактор** | редполитика, продуктовые блоки, язык (орфо/грамм/пункт/стиль) | Финальная вычитка |

**Принцип независимости.** Важное уточнение: «свежий взгляд» получается **сам собой** — каждый запуск этапа это отдельное обращение к модели без памяти о прошлых шагах (модель stateless). Поэтому ценность отдельных проверяющих ролей **не в том, что это «другой агент»**, а в трёх вещах:
1. **Разный мандат (инструкция):** проверяющему задаётся придирчивая установка «сверяй и сомневайся», автору — «пиши убедительно». Один общий промт «напиши и заодно проверь» делает модель склонной соглашаться сама с собой.
2. **Разный вход:** проверяющему можно подать только текст и источники, без авторского «замысла», чтобы он судил по фактам, а не по намерению.
3. **Разные знания:** у каждой роли своя «методичка».

Поэтому проверяющие роли (Фактчекер, SEO, Редактор) отделены от Автора. Для медицинского контента строгая независимая проверка особенно важна.

Опциональный этап «Стиль/Главред» закреплён за ИИ-редактором (стиль и язык — его зона); отдельная роль «Главред» не выделяется.

## 3. Статусная модель темы (упрощённая)

**Нет жёсткого набора из десятка статусов.** Статус вычисляется автоматически из факта «докуда дошли» + текущая метка.

### Из чего складывается статус
1. **База:** «последний успешно завершённый этап» + дата. Пример: `Семантика-в-текст · 20.05`.
2. **Индикатор семантики** (отдельно): `семантика: 24 запроса` или `семантика: не собрана`.
3. **Наложенная метка** (динамическая):
   - ⏳ `идёт: [этап]` — пока этап выполняется
   - ⚠ `ошибка: [этап]` — если последний запуск упал (прогресс не сбрасывается)
   - ⏸ `пауза` — редактор отложил тему вручную
   - ✓ `опубликовано · [дата]` — после публикации

### Что это даёт
- Редактор в Контент-плане видит одной строкой: докуда дошла тема, когда, собрана ли семантика, не висит ли ошибка.
- Нет искусственных переходов и «застрявших» статусов.

### Переходы
- **Этап успешно завершён** → «последний успешный этап» = этот этап, дата обновляется. (Если запустили этап «не по порядку» — отметка обновляется на фактически последний успешный; порядок не навязывается.)
- **Этап упал** → метка `ошибка: [этап]`. База не меняется.
- **Пауза** → метка `пауза`, снимается при следующем запуске.
- **Публикация** (только явный клик) → метка `опубликовано · дата`.
- **«Вернуть в работу»** после публикации → метка снимается, лента версий продолжается.

## 4. Главные пользовательские разделы приложения

Структура: **сквозной блок этапов** (виден всегда) + 5 основных разделов (Контент-план, Рабочее пространство темы, Очередь, Шаблоны, База знаний клиники) + расширенный режим.

### 4.0 Сквозной блок этапов (всегда виден)
Этапы — **постоянный элемент интерфейса**. Показывает для открытой темы цепочку этапов с состоянием (готово / идёт / ошибка / пропущен). Сгруппированы:
- **Подготовка:** Бриф, Семантика
- **Создание:** Черновик, Продуктовые блоки (опц.)
- **Оптимизация:** Семантика-в-текст, Стиль (опц.)
- **Контроль:** Проверка SEO, Фактчекинг, Финальная вычитка
- **Выпуск:** Публикация

Клик по этапу — переход к нему. Любой этап можно запустить в любой момент. Ручная правка доступна на любом этапе (см. раздел 7).

### 4.1 Контент-план (главный экран)
Главная точка входа — список всех тем.
- Колонки: ID, название, тип, статус (последний этап + дата + метка), индикатор семантики, метка очереди, **стоимость по теме**
- Фильтры: по этапу, типу, **направлению, врачу**, «в очереди», «с ошибкой», «опубликовано», поиск
- Кнопки: «новая тема», «добавить выбранные в очередь», «повторить упавшие»
- Компактный виджет очереди в левом сайдбаре (см. 4.3)

### 4.2 Рабочее пространство темы (side-by-side как «полотно темы»)
Главный экран продукта. Основной режим — **две колонки**, постоянный центр, пока вы работаете с темой (**вариант X — «полотно темы»**):
- **Левая колонка** — текущая версия текста.
- **Правая колонка** — новая версия (результат этапа) с **подсвеченными изменениями**; здесь же доступна ручная правка фрагментами или целиком (см. раздел 7).
- **Переключение этапов** меняет текст в тех же колонках с минималистичной анимацией. Колонки не закрываются.
- Очередь, Шаблоны, База знаний — отдельные разделы; при переходе туда полотно темы временно скрывается.
- **Кнопки принятия:** «принять всё → следующий этап», «принять частично» → следующий этап, «отклонить».
- **Шапка:** название, тип, объём, ID, статус, ссылка на Docs (если опубликовано), кнопки «запустить этап», «в очередь», «пауза».
- **Лента версий** (правый или выдвижной сайдбар): два вида — «по времени» и «по этапам»; выбор двух версий → side-by-side diff; кнопка «сделать текущей» (откат).
- **Лог темы** (нижняя выдвижная панель): что и какой агент делал, когда, успех/ошибка.

Когда нет «новой версии» для сравнения — правая колонка показывает редактор для ручной правки текущего текста.

### 4.3 Очередь и automation (компактно в сайдбаре + полный раздел)
Очередь живёт **компактно в левом сайдбаре**: текущая тема, текущий этап, мини-прогресс, кнопки «стоп после текущей / стоп немедленно». Полный раздел: список тем (drag-перетаскивание), сценарий у каждой, лог (фильтр «только ошибки»), кнопки «старт / стоп немедленно / остановить после текущей / сброс / повторить упавшие». Подробно — раздел 8.

### 4.4 Шаблоны
Категории (понятные названия):
- **Общие правила** — технические инструкции для всех агентов (русский, markdown, не выдумывать). Вшиты, меняются редко.
- **Редполитика и источники** — редполитика, работа с источниками, SEO-рекомендации (редактируемые тексты; это «скрытый контекст» из 2.6).
- **ИИ-роли (агенты)** — их контекст и закреплённые этапы (см. 2.14).
- **Промты этапов** — по типам статей, с переменными `{{…}}` и историей версий.
- **Продуктовые блоки** — каждый блок отдельный шаблон.
- **Скиллы** — мини-промты для правки выделенного фрагмента (см. 2.8).
- **Типы статей** — создать/редактировать.
- **Песочница** — прогон шаблона на теме без сохранения как текущей версии.
- **Реестр переменных** — список доступных переменных с источниками (см. 2.5); вставка `{{…}}` в промт.

Открывается редко, не мешает ежедневной работе.

### 4.5 Расширенный режим (скрыт по умолчанию)
Доступ: иконка настроек (шестерёнка) → тумблер «Показать расширенные функции». По умолчанию выключен.
- Параметры моделей шаблонов (модель, температура, max tokens)
- Импорт/экспорт шаблонов
- Полные логи запусков с расходом токенов и стоимостью

### 4.6 База знаний клиники
Отдельный раздел с **древовидным редактором** справочника (см. 2.13): создание / перемещение / переименование узлов, наполнение содержимым, указание ссылок-источников у направлений. Есть **фильтрация/срезы**. Отсюда узлы прикрепляются к темам.

## 5. Этапы пайплайна (сценарий по теме, end-to-end)

Здесь определяются **этапы** — действия над темой. Это же — сквозной сценарий работы.

**Что такое этап.** Логический шаг работы: знает вход и выход (новую версию в ленте), можно ли пропустить, какой он — автоматический, ручной или гибридный, и **какой агент его выполняет** (см. 2.14). **Пайплайн — рекомендованный порядок, а не жёсткие рельсы:** по умолчанию любой этап можно запустить в любой момент.

**Ручная правка — сквозная, а не отдельный этап.** Редактировать текст (выделять фрагменты, регенерировать их, применять пресеты, фиксировать версии) можно на любом этапе в правой колонке — подробно в разделе 7.

Видимых этапов **одиннадцать: девять обязательных** (Бриф, Семантика, Структура, Черновик, Семантика-в-текст, Проверка SEO, Фактчекинг, Финальная вычитка, Публикация) **+ два опциональных** (Продуктовые блоки, Стиль). Плюс режим «Быстрая проверка» вне пайплайна.

### Шаг 0. Бриф темы (ручной)
В Контент-плане «новая тема». Бриф — **фиксированная структурированная форма**:
- Название *(обяз.)*
- Тип статьи *(обяз.)*
- **Направление** — выбор из Базы знаний, напр. «Лучевая терапия» *(обяз. перед Черновиком)*
- **Врач** — выбор из Базы знаний, напр. «Усычкин С.В.» *(опц.)*
- Целевой объём
- Флаги (Стиль вкл/выкл)
- Заметки *(опц.)*

«Направление» и «Врач» выбираются из узлов Базы знаний (см. 2.13) и **автоматически прикрепляют** эти узлы к теме (можно добавить и другие узлы вручную — система их предложит). Эти же поля используются для фильтрации в Контент-плане. **Запуск Черновика заблокирован**, пока не заполнены название, тип и направление.

### Шаг 1. Семантика (ручной + опц. Топвизор)
Добавление запросов (пачкой), таблица с частотностью и статусами (вкл/выкл/обязательный), опц. рекомендации и авто-исключение нерелевантных. Опционально — **импорт кластеров и частотностей из Топвизора**. Индикатор «семантика собрана» появляется в статусе темы.

### Шаг 2. Структура (агент: ИИ-автор)
Один клик «сгенерировать структуру». Агент строит план статьи — иерархию H1/H2/H3 с краткими пометками по разделам, **без ключевых запросов**. План хранится как **редактируемое поле темы** (по образцу семантики) и правится вручную в окне «Структура»; отдельной версии в ленте он не создаёт. В Черновик план подставляется через переменную `{{структура}}`. В контекст агента **скрыто подставляются** редполитика и источники (см. 2.14).

### Шаг 3. Черновик (агент: ИИ-автор)
Один клик «сгенерировать черновик». Агент пишет текст **по утверждённой структуре** (`{{структура}}`), сохраняя заголовки H1/H2/H3; если план пустой — продумывает структуру сам. В его контекст **скрыто подставляются** редполитика, источники, SEO-рекомендации и правила типа статьи. Результат — версия в ленте с меткой «черновик».

### Шаг 4. Продуктовые блоки (опционально, агент: ИИ-автор)
Если для типа статьи есть блоки — редактор открывает конструктор: отмечает галочками нужные блоки, агент генерирует их (данные из Базы знаний, см. 2.7, 2.13) и встраивает в текст. Результат — новая версия. Если блоки не нужны — шаг пропускается.

### Шаг 5. Семантика-в-текст (агент: ИИ-автор)
Один клик «встроить семантику». Агент естественно встраивает запросы (учёт обязательных/частотности, без порчи русского), дописывает/правит H1, генерирует Title и Description. Результат — side-by-side. В конце агент **списком отчитывается**, какие запросы встроил и в чём не уверен — это лёгкий ориентир, а не строгая проверка. Плюс предупреждения о длине Title/Description.

### Шаг 6. Стиль / «Главред» (опционально, агент: ИИ-редактор)
Включается флагом темы. Создаёт **параллельную стилистическую версию**, side-by-side с подсветкой. Принятие всё / выборочно по абзацам / отклонить. Жёсткое правило: не меняет **структуру — H1/H2/H3** (помечается, по умолчанию исключается из принятия). Содержательные правки — на усмотрение редактора (приложение не отделяет медицинские факты само).

### Шаг 7. Проверка на соответствие SEO (агент: ИИ-SEO)
Сравнивает текст с SEO-рекомендациями (заголовки, ключи, Title/Description, структура, объём, плотность, рекламность). **side-by-side с предложенными правками** + список замечаний с местами. Принятие целиком/точечно или ручная правка. Размещение: после «Семантика-в-текст» и «Стиль» (поймать, если стилистика выкинула ключ).

### Шаг 8. Фактчекинг (агент: ИИ-фактчекер)
Сверяет факты в тексте с источниками и **Базой знаний** (см. 2.13): имена врачей, названия процедур, цены, доказательные утверждения. Результат — side-by-side с предложениями + список замечаний с местами; расхождение («в тексте 5000 ₽, в справочнике 4500 ₽») попадает в замечания. Принятие точечно. Фактчекер — **не автор**, поэтому проверяет свежим взглядом.

### Шаг 9. Финальная вычитка (агент: ИИ-редактор)
Язык и соответствие редполитике: орфография, грамматика, пунктуация, стилистика и канцеляризмы, перегруженные предложения, «вода». **Принцип:** ИИ не переписывает молча — показывает side-by-side + список замечаний по категориям с количеством, местами и пояснениями. Редактор принимает правки целиком или точечно; последнее слово за человеком. Когда замечания закрыты — текст готов к публикации.

### Шаг 10. Публикация (ручной)
Один клик «опубликовать» → подтверждение с превью → создаётся документ в Google Docs, в теме сохраняется ссылка и дата (сущность «Внешний документ», см. 2.12), метка `опубликовано`. «Вернуть в работу» продолжает ленту версий. Подробно — раздел 9.

### Режим «Быстрая проверка» (вне пайплайна)
Отдельное окно для разовых задач. Редактор **вставляет текст из буфера**, выбирает одну проверку (фактчек / язык-ошибки / проверка SEO / стиль) — её выполняет соответствующий агент, результат как side-by-side с правками. Без создания темы. Результат можно **скопировать в буфер** или **сохранить как тему**. По умолчанию не хранится.

## 7. Как работает сквозная ручная правка

Ручная правка — это **не отдельный этап, а сквозная возможность** на любом этапе.

- **Где:** в правой колонке любого этапа (или в редакторе текущего текста, когда сравнивать не с чем).
- **Инструменты:**
  - Выделить **любой фрагмент** (границы выбираете сами) → «регенерировать этот фрагмент» с указанием, что не нравится (помогает ИИ-автор).
  - Выделить фрагмент → применить **пресет «скилл»** (см. 2.8: «перепиши в инфостиле», «упрости», «уточни», «убери канцелярит») → локальный side-by-side → принять.
  - Мягкие подсказки: длинные предложения, повторы, явные канцеляризмы (не блокеры). Формируются **алгоритмически в приложении, без обращения к ИИ** — мгновенно, как подчёркивание в текстовом редакторе: длина предложения по порогу, повторы однокоренных слов рядом, штампы по редактируемому **словарю канцеляризмов**. Стартовый словарь генерируется ИИ и дальше пополняется вручную (в т.ч. под медтематику); проприетарные базы (Главред/Тургенев) не копируются. Подсказки не дублируют этап «Финальная вычитка»: они грубые и мгновенные, а ИИ-вычитка — глубокая и в конце.
  - «Зафиксировать как новую версию».
- **Каждое явное действие — новая версия** в ленте с пометкой источника. **Явное действие** — это осознанная команда, порождающая снимок текста: успешный запуск этапа, принятие правок (целиком/частично), «зафиксировать как новую версию», применение пресета, откат, импорт из Docs. **Не создаёт версию:** черновое редактирование в поле (рабочее состояние автосохраняется, но в ленту не попадает), просмотр, переключение этапов, промежуточные подшаги внутри одного запуска. Так лента остаётся чистой историей значимых точек, а не записью каждого нажатия клавиши.
- **Принятие частично:** по фрагментам или по отдельным замечаниям; создаётся гибридная новая версия.
- **Откат:** выбор старой версии → «сделать текущей» → запись «откат к версии X», история сохранена.
- **Сравнение версий:** выбрать две версии из ленты → side-by-side diff.
- **Подтверждение финала:** после «Финальной вычитки» кнопка «готово к публикации» помечает текущую версию утверждённой и включает «опубликовать».

## 8. Как работает очередь и фоновая automation

- **Модель:** строго последовательная (одна тема за раз) — предсказуемость и контроль расходов.
- **Добавление:** из Контент-плана, из темы, из раздела «Очередь».
- **Сценарии:** «до черновика», «до семантики-в-текст», «до проверок», «полный» (стоп перед публикацией), **свободный выбор этапов**. Полный пайплайн всегда останавливается перед публикацией.
- **Прогресс:** компактно в сайдбаре + полный лог в разделе.
- **Остановка:** «остановить после текущей» (мягко), «стоп немедленно» (обрывает запуск, тема возвращается в состояние до этапа), «сброс» (с подтверждением).
- **Ошибки:** упавшая тема получает метку `ошибка`, очередь **не встаёт**, идёт к следующей.
- **Повторить упавшие:** массовая операция в Контент-плане и в разделе «Очередь».
- **Без открытия темы доступно:** постановка/удаление из очереди, смена сценария, просмотр ошибок, «повторить упавшие», порядок.

---

## 9. Как работает публикация и внешний документ

- **Модель:** Google Docs — только финальная публикация. Все правки, версии и diff — внутри приложения.
- **Почему так:** убирает главную старую боль — двустороннюю синхронизацию. Истина всегда в приложении.
- **Возврат правок из Docs:** только вручную — copy-paste в новую версию с меткой «импорт из Docs, дата».
- **Diff:** только внутри приложения, между любыми версиями.
- **Ничего не теряется:** все версии и логи навсегда (можно архивировать, не удалять). Список всех публикаций с датами (сущность «Внешний документ», см. 2.12).
- **Повторная публикация:** по умолчанию «создать новый документ»; опция «перезаписать» — с подтверждением.

## 10. Что было неудобно в старом приложении

1. Слишком много видимых этапов (редполитика, источники, структура — без наблюдаемого артефакта).
2. Двусторонняя синхронизация с Google Docs — конфликты «какая версия настоящая».
3. Размытость артефактов — промежуточные технические сущности вместо «текст и его версии».
4. Размытость статусов — не было ответа «что от меня нужно прямо сейчас».
5. Очередь без явной модели поведения при ошибках.
6. Правила и шаблоны мешали ежедневной работе.
7. Главред мог переписать текст без возможности отката.
8. Не было лёгкого сравнения версий.
9. Не было ясного разделения «что делает система» и «что я редактирую».
10. Слабый лог.

## 11. Что предлагаем улучшить (конкретные решения)

1. **Единая лента версий** вместо отдельных слотов.
2. **Упрощённый статус**: последний успешный этап + дата + индикатор семантики + метки «идёт/ошибка».
3. **Side-by-side как основной экран** с подсветкой и принятием целиком/частично.
4. **Сквозной блок этапов**, видимый всегда, с группировкой.
5. **Произвольный запуск этапов по умолчанию**.
6. **Очередь компактно в сайдбаре** + три кнопки остановки + «повторить упавшие».
7. **Режим «Быстрая проверка»** без создания темы.
8. **Этап «Проверка на соответствие SEO»**.
9. **Скрытый контекст** (редполитика, источники, SEO) вместо видимых этапов.
10. **Этап-конструктор «Продуктовые блоки»**.
11. **Проверки как side-by-side + список замечаний**, принятие точечно; ИИ не переписывает молча.
12. **Главред — параллельная версия** с защитой структуры H1/H2/H3.
13. **Пресеты «скиллов» для фрагментов**.
14. **Регенерация выделенного фрагмента**.
15. **Откат без потери истории**; версии не удаляются, только архивируются.
16. **Публикация без двусторонней синхронизации**.
17. **Песочница шаблонов**.
18. **Расширенный режим** скрывает параметры моделей и т.п.
19. **Мягкие подсказки** при правке.
20. **Семантика-в-текст**: лёгкий отчёт ИИ, без алгоритмической проверки вхождения.
21. **База знаний клиники** — внутренний древовидный справочник: прикрепление узлов, ссылки-источники, фильтрация.
22. **Фактчекинг по справочнику**.
23. **Единая лента версий с двумя видами**.
24. **Side-by-side как «полотно темы»** с анимацией.
25. **Опциональный импорт семантики из Топвизора**.
26. **Стоимость по теме в Контент-плане**.
27. **ИИ-роли (агенты)** с собственным контекстом; строгость проверки — за счёт разных мандатов, входа и знаний у проверяющих ролей (а не за счёт «другого агента»: свежий взгляд и так есть, модель stateless).
28. **Сквозная ручная правка** — отдельный этап «Правка» убран, инструменты доступны на любом этапе.
29. **Финальная проверка разделена** на «Фактчекинг» (фактчекер) и «Финальную вычитку» (редактор).

## 12. Что можно убрать без потери ценности

1. Видимые этапы «редполитика», «источники», «изучение SEO» → контекст ИИ-ролей.
2. Этап «структура» как отдельный шаг → внутренний подшаг «Черновика».
3. Отдельные слоты-артефакты → единая лента версий.
4. Жёсткий набор статусов → вычисляемый статус.
5. Двусторонняя синхронизация с Google Docs и diff с Docs → удалить полностью.
6. «Workspace правил» в ежедневном интерфейсе → в «Шаблоны».
7. Автоматическая публикация без явного клика → удалить.
8. Параллельные запуски в очереди → только последовательно.
9. Удаление версий → только архивирование.
10. Молчаливое авто-переписывание текста проверками → только предложения с принятием.
11. **Отдельный этап «Правка»** → сквозная ручная правка на любом этапе.

## 13. Что в «расширенном режиме» / для power-user

- Параметры моделей шаблонов (модель, температура, max tokens)
- Импорт/экспорт шаблонов
- Полные логи запусков с токенами и стоимостью
- Создание и полная настройка новых типов статей и ИИ-ролей
- Песочница шаблонов
- Резервное копирование: локальный экспорт всей базы + опц. авто-бэкап

> Важно: «свободный выбор этапов» — поведение по умолчанию, не функция power-user.

## 14. Выбранный вариант: B (смелое переосмысление)

Вариант A (оставить все старые видимые этапы, улучшив только UX) **отклонён**: не лечит главную боль.

Принят **Вариант B**:
- 9 обязательных видимых этапов + 2 опциональных.
- Редполитика, источники, SEO-рекомендации — контекст ИИ-ролей (структура вынесена в видимый редактируемый этап).
- ИИ-роли (агенты) с независимой проверкой.
- Единая лента версий, упрощённые статусы, side-by-side как основной экран.

Условие: один раз перенести содержимое старых видимых этапов в контекст ИИ-ролей. Разовая работа, дальше окупается.

## 15. Обоснование выбора

1. Главная боль — «слишком много видимых этапов». Вариант B её решает.
2. Single-user → переходный барьер несущественен.
3. Скрытые этапы не теряются (в контексте ролей и «Шаблонах»).
4. Старые промты становятся контекстом ролей или подэтапами.
5. Вариант B лучше масштабируется.

## 16. Открытые вопросы — статус

Закрыто (версии 3–7):
- **Метрики «Семантика-в-текст»** → лёгкий отчёт ИИ (раздел 5).
- **Стоимость запусков** → Контент-план (4.1).
- **«База знаний клиники»** → внутренний древовидный справочник (2.13, 4.6).
- **Версии текста** → единая лента + два вида (2.3).
- **Навигация side-by-side** → «полотно темы» (4.2).
- **Топвизор** → опциональный импорт на «Семантике» (2.2).
- **Промежуточная зона согласования** → не нужна.
- **Резервное копирование** → локальный экспорт + опц. авто-бэкап (13).
- **Быстрая проверка** → копировать/сохранить как тему; не хранится (раздел 5).
- **Внешняя SEO-аналитика** → для старта не нужна.
- **Сущности vs этапы** → раздел 2 = сущности, раздел 5 = этапы (версия 6).
- **Этап «Правка»** → убран, правка сквозная (версия 7).
- **ИИ-роли (агенты)** → введены, 4 агента, независимая проверка (2.14, версия 7).
- **«Финальная проверка»** → разделена на «Фактчекинг» и «Финальную вычитку» (версия 7).

Закрыто (версия 7.3):
- **Агент «Стиль/Главред»** → остаётся за ИИ-редактором; отдельная роль не выделяется (2.14).
