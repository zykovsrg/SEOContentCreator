# Design: Под-проект «Публикация в Google Docs»

Дата создания: 2026-06-21
Статус: draft, ждёт ревью пользователя
Родительская спека: `docs/superpowers/specs/2026-05-19-content-system-redesign-design.md` (раздел 9, сущность «Внешний документ» 2.12, этап «Выпуск → Публикация» 4.0)
Автор обсуждения: zykovsrg + AI product architect

Это дизайн отдельного под-проекта — финального этапа пайплайна. Закрывает путь «от идеи до выпуска»: принятая версия статьи публикуется в Google Docs. Документ описывает логику, компоненты, данные и обработку ошибок; точные сигнатуры и файловую раскладку определит план реализации.

## Ключевые решения (зафиксированы при brainstorming 2026-06-21)

- **Способ публикации:** настоящая интеграция с Google API (вариант A), а не ручной экспорт.
- **OAuth:** свой, на `URLSession`, без сторонних библиотек (вариант A) — проект держится без зависимостей. Desktop-флоу с перехватом ответа на `127.0.0.1` и PKCE.
- **Содержимое документа (v1):** только текст — заголовки H1/H2/H3, абзацы, жирный/курсив, маркированные и нумерованные списки. **Изображения не вставляем** (вставляются вручную).
- **Ключи Google:** Client ID и Client Secret вводятся в Настройках и хранятся в Keychain (как ключ OpenAI). В код и git не попадают.
- **Размещение:** документы кладутся в отдельную папку приложения в Google Drive (имя по умолчанию — «SEO-статьи клиники»; находится по имени или создаётся при первой публикации).
- **Повторная публикация:** по умолчанию «создать новый документ»; «перезаписать существующий» — только с подтверждением.

## Инварианты (из родительской спеки, не нарушать)

- Google Docs — **только финальная публикация**, без обратной синхронизации.
- Публикация — **только по явному клику** «Опубликовать».
- Возврат правок из Docs — только вручную (copy-paste в новую версию с меткой «Импорт из Docs»; источник версии `importFromDocs` уже есть).
- Ничего не теряется: все публикации с датами хранятся.

## 1. Объём

**Входит (v1):**

- Односторонняя публикация текущей принятой версии статьи (`Topic.currentVersion`) в Google Docs.
- Свой OAuth 2.0 (desktop + PKCE) на `URLSession`.
- Создание документа в отдельной папке Drive (найти существующую по имени или создать).
- Преобразование markdown → Google Docs: H1/H2/H3 как стили заголовков, абзацы, жирный/курсив внутри строки, маркированные и нумерованные списки.
- Повторная публикация: новый документ (по умолчанию) или перезапись существующего (с подтверждением).
- История публикаций темы (сущность «Внешний документ»).
- Ввод и хранение ключей Google в Настройках (Keychain); вход/выход из Google.

**Не входит:**

- Изображения в документе (обложка, иллюстрации) — вставляются вручную; кандидат на будущую доработку.
- Двусторонняя синхронизация с Docs — сознательно никогда.
- Таблицы и выноски-callout (в v1 не нужны; YAGNI).
- Несколько Google-аккаунтов.

## 2. Компоненты

Каждый юнит — одна задача, чёткий интерфейс, тестируется отдельно.

### 2.1 `GoogleAuthService` (`@MainActor @Observable`)

Отвечает за весь OAuth и жизненный цикл токена.

- `signIn()` — формирует URL согласия (скоупы `https://www.googleapis.com/auth/documents` + `https://www.googleapis.com/auth/drive.file`), поднимает локальный HTTP-слушатель на `127.0.0.1:<свободный порт>`, открывает браузер (`NSWorkspace`), перехватывает `code`, обменивает его на access/refresh-токены на `https://oauth2.googleapis.com/token` (с PKCE `code_verifier`), сохраняет токены в Keychain.
- `validAccessToken() async throws -> String` — возвращает свежий access-токен; если истёк — обновляет по refresh-токену.
- `signOut()` — удаляет токены из Keychain.
- Client ID/Secret читает из Keychain (введены в Настройках).
- Состояние для UI: подключён/не подключён, последняя ошибка.

Скоуп `drive.file` даёт доступ **только к файлам, созданным приложением** — приложение не видит остальной Google Drive.

### 2.2 `GoogleDocsClient`

Тонкая обёртка над REST Google через `URLSession` (стиль `OpenAIClient`). Без бизнес-логики.

- `createDocument(title:) async throws -> docID` — `POST https://docs.googleapis.com/v1/documents`.
- `batchUpdate(docID:, requests:) async throws` — `POST .../documents/{id}:batchUpdate`.
- `clearBody(docID:) async throws` — удаляет тело документа (для перезаписи).
- `findOrCreateFolder(name:) async throws -> folderID` — `drive.files.list` по имени + `drive.files.create` при отсутствии.
- `moveToFolder(fileID:, folderID:) async throws` — `drive.files.update` (`addParents`).
- Все вызовы — через обёртку с повторными попытками и экспоненциальным backoff на временные сбои (по образцу прошлой реализации).
- Access-токен получает через `GoogleAuthService.validAccessToken()`.

### 2.3 `MarkdownToDocsConverter`

Чистая функция, без сети и без состояния: markdown текущей версии → упорядоченный список запросов `batchUpdate`.

- Распознаёт: `#`/`##`/`###` → стили `HEADING_1/2/3`; абзацы → `NORMAL_TEXT`; `- `/`* ` и `1. ` → маркированные/нумерованные списки; `**…**`/`*…*` → жирный/курсив по диапазонам (`updateTextStyle` с вычисленными индексами).
- Главный объект для юнит-тестов: вход — строка markdown, выход — детерминированная структура запросов.

### 2.4 `ArticlePublisher` (`@MainActor @Observable`)

Оркестратор сценария публикации. Зависимости (auth, client, конвертер) внедряются — как в `StageExecutor` — чтобы тестировать с моками.

- `publish(topic:, mode:) async` где `mode = .newDocument | .overwrite`.
- Шаги: получить токен → найти/создать папку → создать документ (или для перезаписи переиспользовать сохранённый `docID` + `clearBody`) → конвертер по `topic.currentVersion.text` → `batchUpdate` → переместить в папку → сохранить `ExternalDocument` и обновить `Topic.externalDocURL`/`publishedAt`.
- Состояние: `isPublishing`, `lastErrorMessage`.

## 3. Модель данных

Новая SwiftData-модель **`ExternalDocument`**:

- `docURL: String`, `docID: String`, `publishedAt: Date`, `mode` (новый/перезапись), связь `topic: Topic`.
- У `Topic` — новая связь `publications: [ExternalDocument]` (`@Relationship`, cascade) для истории.
- Существующие `Topic.externalDocURL` и `Topic.publishedAt` сохраняются как «последняя публикация» (для шапки и статуса `published` в `TopicStatus`). Лёгкая миграция SwiftData (добавление модели и связи).

## 4. Интерфейс и поток

1. **Кнопка.** Этап «Выпуск → Публикация» в сквозном блоке этапов. Это действие, а не ИИ-генерация — клик открывает **панель публикации**.
2. **Превью-подтверждение.** Заголовок документа (= название темы), какая версия публикуется, целевая папка Drive; кнопка «Опубликовать». Для уже опубликованной темы — выбор «создать новый» / «перезаписать существующий» (перезапись с дополнительным подтверждением).
3. **Результат.** Документ создан/обновлён; в тему сохранены ссылка и дата; статус «Опубликовано»; ссылка на Docs видна в шапке темы.
4. **История.** Список всех публикаций темы с датами и ссылками (из `Topic.publications`) — в рабочем пространстве темы.
5. **Настройки.** Раздел Google: поля Client ID и Client Secret (Keychain), кнопка «Войти в Google» / «Выйти», индикатор «Подключено».

## 5. Обработка ошибок

- Нет Client ID/Secret → «Укажите ключи Google в Настройках», публикация не стартует.
- Вход отменён или доступ не выдан → сообщение; статус темы не меняется.
- Refresh-токен отозван, обновление не удалось → предложить войти заново.
- Сетевые/серверные ошибки API → повтор с backoff; при финальном провале — человекочитаемое сообщение, тема **не** помечается опубликованной, запись в лог.
- `docID` сохраняется сразу после создания документа: повторная попытка после сбоя форматирования перезаписывает тот же документ, а не создаёт пустые дубликаты.

## 6. Тестирование

TDD, Swift Testing — как в остальных под-проектах.

- **`MarkdownToDocsConverter`** — основной набор юнит-тестов: заголовки H1/H2/H3, абзацы, маркированные/нумерованные списки, жирный/курсив (в т.ч. вложенный и на границах), пустые строки, краевые случаи индексов.
- **`GoogleDocsClient`** — мок `URLProtocol` (как в тестах OpenAI): форма HTTP-запросов (create/batchUpdate/folder/move), backoff на временных ошибках, маппинг ошибок в человекочитаемый текст.
- **`ArticlePublisher`** — с подставными `auth`/`client`/конвертером: ветки «новый документ» против «перезапись», сохранение `ExternalDocument`, обновление полей `Topic`, все ветки ошибок (нет ключей, не выполнен вход, сбой API).
- **`GoogleAuthService`** — логика обновления токена на моке `URLSession`; обмен кода на токен. Браузерный вход и перехват на `127.0.0.1` — интеграционная часть, проверяется вручную.
- **Ручные проверки:** один реальный вход в Google; реальная публикация; повтор (новый + перезапись); попадание документа в нужную папку; формат документа (заголовки/списки/выделения) визуально.

## 7. Будущие доработки (вне объёма)

- Вставка изображений (обложка, иллюстрации) в документ — требует заливки на Drive и позиционирования.
- Таблицы/выноски, если появятся в шаблонах.
- Настраиваемое имя/структура папок Drive.
