# Design: ИИ-роли и блоки контекста (раздел «Шаблоны», срез B)

Дата: 2026-05-23
Под-проект: 6 (срез B раздела «Шаблоны»)
Статус: согласован, готов к плану

## 1. Цель

Сделать ИИ-роли (агентов) и их «методички» первоклассной редактируемой сущностью.

Сейчас:
- роли как сущности нет — имя агента жёстко зашито за этапом в коде (`PipelineStage.agentName`);
- «методичка» роли (идентичность «кто ты», редполитика, принципы источников, SEO-рекомендации) вшита прямо в системные промты этапов (`StageTemplateDefaults`), с дублированием между этапами.

Срез B выносит это в редактируемые **блоки контекста** (общие, переиспользуемые) и **ИИ-роли** (ссылаются на блоки + свой мандат) и подставляет контекст роли в системный промт автоматически. Это убирает дубли, даёт правку «в одном месте» и ложится на спеку §2.6, §2.14, §4.4.

## 2. Объём

В объёме:
- Сущность «Блок контекста» (редактируемый текст). Стартовые: Редполитика, Принципы работы с источниками, SEO-рекомендации.
- Сущность «ИИ-роль» (4: Автор, SEO, Фактчекер, Редактор) — редактирование имени, «установки» (мандата) и набора используемых блоков.
- Автоматическая сборка системного промта из роли + блоков при запуске этапа.
- Перенос методичек из промтов этапов в блоки/роли + укорачивание дефолтов этапов.
- Авто-миграция существующих промтов этапов (только системный промт).
- UI: в разделе «Шаблоны» — категории «ИИ-роли» и «Редполитика и источники» + редакторы.

Вне объёма (отдельные под-проекты):
- Создание/удаление ролей и переназначение ролей за этапами.
- Срезы C (типы статей), D (продуктовые блоки), E (скиллы/песочница).
- Категория «Общие правила» (§4.4) — пока остаётся как есть.
- Полное версионирование с историей блоков/ролей (только счётчик версии + дата).

## 3. Модель данных

### 3.1 ContextBlock (новый @Model)
- `uuid: UUID`
- `key: String` — устойчивый идентификатор: `editorialPolicy`, `sources`, `seoGuidelines`
- `title: String` — отображаемое имя («Редполитика»)
- `text: String` — редактируемое содержимое
- `version: Int` — счётчик (растёт при сохранении)
- `createdAt: Date`, `updatedAt: Date`

### 3.2 AIRole (новый @Model)
- `uuid: UUID`
- `key: String` — устойчивый: `author`, `seo`, `factChecker`, `editor`
- `name: String` — отображаемое имя («ИИ-автор»)
- `mandate: String` — «установка роли» (кто ты + как работаешь)
- `blockKeys: [String]` — ключи используемых блоков (членство). Порядок склейки в промте — канонический (фиксированный порядок блоков из `ContextBlockDefaults`: editorialPolicy → sources → seoGuidelines), не зависит от порядка в массиве
- `version: Int`, `createdAt: Date`, `updatedAt: Date`

Связь роль↔блок хранится как массив ключей `blockKeys` (просто, надёжно, легко засевать и переключать галочками). Блоки резолвятся по `key`.

### 3.3 Связь этап→роль (в коде, фиксирована)
`PipelineStage` получает `roleKey: String`:
- `draft`, `productBlocks`, `semanticsInText` → `author`
- `seoCheck` → `seo`
- `factCheck` → `factChecker`
- `finalReview` → `editor`

Существующее `agentName` остаётся дефолтным именем (используется при засеве и как фолбэк). В UI имя агента берётся из `AIRole.name` по `roleKey` (фолбэк — `agentName`).

## 4. Сборка системного промта

Чистый порядок:
```
[mandate роли]

[текст блока 1]

[текст блока 2]
…
[системный промт этапа — если задан]
```

- **RoleContextAssembler** (чистая функция): вход — роль + её блоки (в каноническом порядке блоков); выход — строка контекста (mandate + тексты блоков, разделённые пустой строкой; пустые части пропускаются).
- **PromptBuilder.build** получает доп. параметр `roleContext: String` (по умолчанию `""`). Возвращает `system = [roleContext, template.systemPrompt]`, склеенные через `\n\n` с пропуском пустых; `user` — без изменений (подстановка переменных как сейчас).
- **StageExecutor**: по `stage.roleKey` находит `AIRole`, резолвит блоки по `blockKeys`, вызывает `RoleContextAssembler`, передаёт результат в `PromptBuilder`. Если роль/блоки не найдены → `roleContext = ""` (генерация не падает, работает только промт этапа).

После укорачивания дефолтов системный промт большинства этапов будет пустым (идентичность + методички уходят в роль/блоки; «что сделать» и так в пользовательском промте). Это нормально; редактор промтов покажет подсказку «системный промт берётся из роли».

## 5. Засев и миграция

### 5.1 Засев новых сущностей (идемпотентно, как `StageTemplateSeeder`)
- Блоки: если нет ни одного `ContextBlock` — засеять 3 из `ContextBlockDefaults` (тексты — извлечённые методички).
- Роли: если нет ни одной `AIRole` — засеять 4 из `RoleDefaults` (имя, mandate, blockKeys).

### 5.2 Укорачивание дефолтов этапов
`StageTemplateDefaults`: системные промты этапов сокращаются — убираем идентичность и методички (теперь в ролях/блоках), оставляем только этап-специфичные дополнения (часто пусто). Пользовательские промты не меняем.

Идентичность/мандаты, переезжающие в роли:
- author: «медицинский редактор-копирайтер; пиши убедительно и достоверно; русский, Markdown»
- seo: «придирчивый SEO-редактор; не переписывай — находи проблемы; сомневайся и сверяй»
- factChecker: «придирчивый медицинский фактчекер; сверяй факты, не выдумывай»
- editor: «придирчивый литредактор-корректор; не переписывай — точечные правки»

Блоки (извлечённые методички):
- Редполитика: достоверность, доказательная осторожность, читабельность, не выдумывать данные, без рекламных преувеличений.
- Принципы работы с источниками: опираться на доказательность; приоритетные источники направления.
- SEO-рекомендации: структура H1/H2/H3; естественность и плотность ключей; корректные Title/Description; объём; без рекламности.

Роль→блоки: author → [editorialPolicy, sources]; seo → [seoGuidelines]; factChecker → [sources]; editor → [editorialPolicy]. (Фактчекер дополнительно получает Базу знаний через переменную `{{база_знаний}}` в пользовательском промте — без изменений.)

Общие технические правила (русский язык, Markdown, «не выдумывать данные»), которые сейчас вшиты в системные промты этапов, при укорачивании НЕ теряются: «русский/Markdown» включаются в mandate каждой роли, «не выдумывать» покрывается блоком «Редполитика». Отдельная редактируемая категория «Общие правила» (§4.4) — вне объёма этого среза.

### 5.3 Авто-миграция существующих промтов этапов (вариант A)
- Хранить `templatesDefaultsVersion` (UserDefaults), текущая = 2.
- При засеве: если сохранённая версия < 2 — у всех существующих `StageTemplate` перезаписать **только** `systemPrompt` на новый короткий дефолт (`userPromptTemplate`, `modelName`, `temperature`, `maxTokens` — НЕ трогать), затем выставить версию = 2.
- Риск: ручные правки именно системного промта этапа будут заменены. Согласовано (важных правок нет). Пользовательский промт / переменные / параметры / новые блоки — не затронуты.

## 6. Интерфейс (раздел «Шаблоны»)

Сейчас `TemplatesView` — мастер-деталь со списком промтов этапов. Добавляется выбор категории слева:
- «Промты этапов» (как есть)
- «ИИ-роли»
- «Редполитика и источники» (блоки)

(Категории — задел под будущие срезы C–E.)

### 6.1 Редактор роли
- Имя (поле)
- «Установка роли» (TextEditor) — мандат
- «Использует блоки» — галочки по списку блоков (вкл/выкл членство)
- «Отвечает за этапы» — показ, только чтение
- «Сбросить к стандартному» (из `RoleDefaults`)

### 6.2 Редактор блока
- Заголовок (показ)
- Текст (TextEditor)
- «Используется ролями» — показ
- «Сбросить к стандартному» (из `ContextBlockDefaults`)

### 6.3 Имя агента из роли
Места показа имени агента этапа (полоса этапов, лог) берут `AIRole.name` по `roleKey` (фолбэк — `PipelineStage.agentName`). Историю (`GenerationJob`/`ArticleVersion.agentName`) не трогаем — это снимки.

## 7. Тестирование (TDD)

- `RoleContextAssembler`: порядок (mandate → блоки в порядке `blockKeys`), разделители, пропуск пустых, отсутствие блока по key.
- `PromptBuilder`: `system` = roleContext + системный промт этапа (склейка, пропуск пустых); подстановка переменных не сломана; пустой roleContext.
- Засев: 3 блока + 4 роли с верными `key`/`blockKeys`; повторный засев не дублирует.
- Укороченные дефолты этапов не содержат методичек.
- Миграция: при версии < 2 `systemPrompt` этапа заменяется коротким, `userPromptTemplate`/параметры сохраняются; версия становится 2; повторный запуск не трогает.
- `PipelineStage.roleKey` корректен для всех этапов.

## 8. Риски и заметки
- SwiftData: добавление 2 новых `@Model` — лёгкое расширение схемы; существующие данные сохраняются. Нужно добавить типы в схему `ModelContainer`.
- `blockKeys` как `[String]` — нет ссылочной целостности на уровне БД; резолв по key с фолбэком (отсутствующий пропускается).
- Запуск юнит-тестов из CLI в этой среде зависает (раннер «hung before establishing connection»); проверка компиляции — `xcodebuild build-for-testing`, прогон тестов — Cmd+U в Xcode.
- Качество текста: содержимое запроса остаётся эквивалентным (идентичность + методички + задача), но без дублей; поведение этапов (включая JSON-замечания проверок) сохраняется.

## 9. Критерий готовности
- Блоки и роли засеяны, редактируются, «Сбросить к стандартному» работает.
- Запуск этапа использует контекст роли (видно по результату).
- Правка блока «Редполитика» влияет на Автора и Редактора.
- Существующие промты этапов мигрированы (короткие), генерация работает.
- Компиляция зелёная; юнит-тесты (Cmd+U) зелёные; smoke пользователем пройден.
