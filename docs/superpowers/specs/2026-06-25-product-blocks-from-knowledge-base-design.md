# Продуктовые блоки из Базы знаний — Design Spec

Date: 2026-06-25
Source future task: FT-20260623-002
Status: approved (user approved 2026-06-25)

## Цель

Заменить жёстко зашитый список из 4 продуктовых блоков в `ProductBlocksSheet`
редактируемыми шаблонами в разделе «Шаблоны». Каждый блок имеет собственный
промт с переменными, которые подставляют данные из прикреплённых узлов Базы
знаний темы. При генерации выбранные блоки встраиваются в текст за один запуск
через существующий `StageExecutor` / стадию «Продуктовые блоки».

## Текущее состояние (до изменений)

- `Views/ProductBlocksSheet.swift` — хардкод `availableBlocks = ["CTA «Записаться»", "Почему мы", "Блок врача", "Преимущества клиники"]`; `onGenerate: ([String]) -> Void` отдаёт имена выбранных блоков.
- `Views/TopicWorkspaceView.swift` — `ProductBlocksSheet(topic:) { runStage(.productBlocks, blocks: $0) }`.
- `Logic/PromptBuilder.swift` — при непустом `selectedBlocks` дописывает в user-промт строку `"Включить продуктовые блоки: " + names.joined(", ")`. Переменные `{{врач_данные}}` (← `topic.doctor.content`) и `{{преимущества}}` (← узлы `topic.attachedNodes` типа `.advantage`) уже подставляются из Базы знаний.
- `Logic/StageExecutor.execute(..., selectedBlocks: [String] = [], ...)` пробрасывает блоки в `PromptBuilder`.
- `Logic/StageTemplateDefaults.swift` — дефолтный промт стадии `.productBlocks` без плейсхолдера блоков.
- Готовый паттерн CRUD-шаблонов: `Models/SkillPreset.swift`, `Logic/SkillPresetDefaults.swift`, `Logic/SkillPresetSeeder.swift`, секция «Скиллы» в `Views/TemplatesView.swift`.

Вывод: зависимость из promotion notes (узлы Базы знаний прикреплены к теме и
доступны в промт-билдере) уже выполнена через `Topic.doctor` / `Topic.attachedNodes`.

## Решения (подтверждены пользователем)

1. **Механизм генерации:** один запуск; промты выбранных блоков вшиваются в
   существующую стадию «Продуктовые блоки».
2. **Модель блока:** название + промт (по образцу `SkillPreset`), без привязки к
   типу узла Базы знаний — переменные уже тянут данные из Базы знаний.

## Дизайн

### 1. Модель данных (новая, аддитивная — миграция не нужна)

Новая SwiftData-сущность `Models/ProductBlock.swift`:

```swift
@Model
final class ProductBlock {
    var uuid: UUID
    var name: String
    var prompt: String
    var order: Int
    var createdAt: Date
    var updatedAt: Date
}
```

- `name` — отображается в `ProductBlocksSheet` и в «Шаблонах».
- `prompt` — текст с переменными (`{{врач_данные}}`, `{{преимущества}}`, `{{направление}}` и т.д.).
- `order` — сортировка списка.

Никаких изменений у `Topic` / `KnowledgeNode` / `StageTemplate` — старые данные в
безопасности. Зарегистрировать `ProductBlock` в SwiftData-схеме приложения там же,
где регистрируются прочие модели (`ModelContainer` / schema).

### 2. Сидирование дефолтов

- `Logic/ProductBlockDefaults.swift` — структура `ProductBlockDefault { name, prompt }` + `all: [...]` (4 дефолта с текущими названиями) + `makeAll()`.
- `Logic/ProductBlockSeeder.swift` — `seedIfNeeded(in:)`: если таблица пуста, вставляет дефолты. Вызвать там же, где `SkillPresetSeeder.seedIfNeeded`.

Дефолты (названия = текущий хардкод), у каждого осмысленный промт:

- `CTA «Записаться»` — призыв записаться на приём.
- `Почему мы` — преимущества клиники из `{{преимущества}}`.
- `Блок врача` — данные врача из `{{врач_данные}}`.
- `Преимущества клиники` — из `{{преимущества}}`.

### 3. Проводка в генерацию (один запуск)

- `ProductBlocksSheet`: список строится из `@Query ProductBlock` (сортировка по `order`), выбор отслеживается по `uuid`. Наружу отдаются **промты** выбранных блоков (а не имена). Сигнатура: `onGenerate: ([String]) -> Void`, где элементы — `prompt` выбранных блоков. (`TopicWorkspaceView` не меняется по сигнатуре.)
- `PromptBuilder.build`: словарь подстановок применяется и к шаблону стадии, **и к каждому промту блока**. Собранные (после подстановки) промты блоков объединяются и вставляются:
  - если в шаблоне стадии есть плейсхолдер `{{продуктовые_блоки}}` — подставляются туда;
  - иначе (старый сохранённый шаблон без плейсхолдера) — **fallback**: дописываются в конец user-промта под заголовком. Так старые установки не ломаются.
- `Logic/StageTemplateDefaults.swift`: в дефолтный промт стадии `.productBlocks` добавить `{{продуктовые_блоки}}` (для свежих установок и кнопки reset).

Сигнатуры `StageExecutor.execute(..., selectedBlocks: [String], ...)` и
`PromptBuilder.build(..., selectedBlocks: [String], ...)` не меняются по типу —
меняется только смысл содержимого (тексты промтов вместо имён).

### 4. Редактирование в «Шаблонах»

Новая секция «Продуктовые блоки» в `Views/TemplatesView.swift` по паттерну
секции «Скиллы»: список с `@Query`, добавить / редактировать / удалить, кнопка
сброса к дефолтам (как у Скиллов).

### 5. Тесты

- `PromptBuilder`:
  - промты выбранных блоков попадают в user-промт;
  - переменные Базы знаний (`{{врач_данные}}`, `{{преимущества}}`) внутри промта блока подставляются;
  - вставка через `{{продуктовые_блоки}}` при наличии плейсхолдера;
  - fallback-append при отсутствии плейсхолдера;
  - пустой выбор не добавляет ничего.
- `ProductBlockSeeder`: сидирует при пустой таблице; не дублирует при повторном вызове.

## Риски

- Низкие. Новая модель аддитивна; fallback защищает старые сохранённые шаблоны
  стадии.
- Проверка сборки: `xcodebuild build-for-testing` (CLI `xcodebuild test`
  зависает — запуск тестов в Xcode через Cmd+U).

## Вне scope

- Привязка блока к конкретному типу узла Базы знаний (отклонено в пользу
  переменных).
- Изменение схемы `Topic` / `StageTemplate`.
- Песочница/импорт/экспорт шаблонов (отдельная future task FT-20260623-007).
